# Skill 评估与迭代方法（Silver Bullet）

## 目的

本文档定义 `silver_bullet` 中 skill 的评估与迭代方法。目标不是做复杂评测平台，而是建立一个足够稳定、足够轻量的闭环，回答这几个核心问题：

- 这个 skill 是否真的有帮助？
- 它和“不用 skill”相比是否更好？
- 触发是否准确？
- 输出是否稳定？
- 下一轮该改什么？

---

## 一、为什么 skill 需要评估

skill 不是“写完就完”的文档，而是会影响：

- 触发正确率
- 流程稳定性
- 输出质量
- token 消耗
- 用户体验

如果没有评估，常见结果是：

- 描述写得很像样，但根本不触发
- 会触发，但输出质量并没有提升
- 只对一两个例子有效，换个表达就失效
- skill 越写越长，但并没有更强

因此，skill 必须进入“编写 → 测试 → 复盘 → 改写”的迭代闭环。

---

## 二、评估分两类

## 1. 质量评估
看 skill 在真实任务中是否产生更好的结果。

关注：

- 输出是否更完整
- 流程是否更稳定
- 是否减少反复返工
- 是否减少拍脑袋式执行
- 是否更符合预期格式

## 2. 触发评估
看 skill 是否在该触发时触发、不该触发时不触发。

关注：

- 应触发场景是否能命中
- 近邻场景是否误触发
- 描述是否过宽或过窄
- 描述中的关键短语是否足够具体

---

## 三、最小评估闭环

推荐使用下面这个最小闭环：

1. 明确 skill 目标
2. 准备测试 prompt
3. 跑有 skill 的版本
4. 观察结果
5. 必要时与“无 skill”或旧版本对照
6. 记录问题
7. 改 skill
8. 再跑一轮

这就是最小可行的迭代系统。

---

## 四、第一步：先定义“这个 skill 应该变好什么”

在评估前，先回答：

- 这个 skill 想改善什么？
- 它的成功标准是什么？
- 哪些结果是明显失败？

例如：

### 对调试类 skill
成功标准可以是：

- 明确区分现象、证据、假设、验证、下一步
- 不直接跳到修复
- 能识别需要升级到架构层讨论的情况

### 对总控类 skill
成功标准可以是：

- 输出分析、计划、进度结构清楚
- 阶段边界明显
- 可跨会话续接
- 不遗漏验证与归档

---

## 五、第二步：设计测试 prompt

不要只用一句非常干净、非常典型的话去测。  
要准备一组“真实用户会说的话”。

### 推荐数量
每轮先准备 3–5 个质量测试 prompt。  
触发评估可以再单独准备 10–20 个触发样本。

### 质量测试 prompt 的要求
- 尽量真实
- 语气可以自然
- 长短混合
- 不要都长得一样
- 覆盖主要场景与边界场景

### 好的样例特征
- 有上下文
- 有模糊表达
- 有近邻任务
- 有容易误判的描述
- 有不同复杂度

---

## 六、第三步：区分三种对照方式

## 1. 无 skill 对照
适合新 skill。

目标是回答：

> 有这个 skill，是否比默认处理更好？

## 2. 旧版本 skill 对照
适合已有 skill 的优化。

目标是回答：

> 新版是否比旧版更稳、更准、更清楚？

## 3. 单轮人工观察
适合非常轻量的内部 skill。

目标是回答：

> 这版是不是已经明显可用？

不是每次都要做完整对照，但至少要知道当前用的是哪一种。

---

## 七、第四步：观察结果时看什么

不要只看“最终输出像不像”。  
至少看下面 5 件事：

### 1. 触发是否正确
- 该触发时是否使用了 skill
- 不该触发时是否误用了 skill

### 2. 流程是否合理
- 是否遵循 skill 给出的核心步骤
- 是否跳过关键步骤
- 是否在不该扩展时过度展开

### 3. 输出是否满足目标
- 是否更清楚
- 是否更完整
- 是否更接近预期结构
- 是否有明显漏项

### 4. 成本是否可接受
- 是否引入大量无效啰嗦
- 是否明显增加上下文负担
- 是否为很小的问题动用了过重流程

### 5. 稳定性如何
- 换个 prompt 说法，是否还能维持表现
- 是否只对一个例子表现好

---

## 八、如何记录问题

每轮评估后，不要只说“还行”或“不太好”。  
应该至少记录这三类问题：

## 1. 触发问题
例如：
- 描述太宽，普通 bugfix 也会误触发
- 描述太窄，用户没说关键词就不触发

## 2. 流程问题
例如：
- 先输出结论，没有采证
- 太早进入规划
- 忘记回归验证
- references 提示不清楚

## 3. 内容问题
例如：
- 缺少输出模板
- 没明确不适用场景
- 规则太抽象
- SKILL.md 过胖

---

## 九、如何改 skill：按症状下药

## 1. 触发不准
优先改：

- frontmatter 的 `description`
- 明确增加应触发短语
- 补充不适用场景
- 收窄模糊词

## 2. 正文太胖
优先改：

- 把长内容迁到 `references/`
- 只保留高频必读内容在 `SKILL.md`

## 3. 流程不稳
优先改：

- 增强阶段说明
- 增加输出格式
- 增加完成门/验证门
- 减少模糊表述

## 4. 重复劳动太多
优先改：

- 把固定逻辑抽到 `scripts/`
- 增加 `examples/`
- 增加 checklist

---

## 十、如何评估 description 的触发质量

这是非常关键的一块。

### 设计两类样本

#### A. 应触发样本
这些 prompt 明确应该使用 skill。

要求：
- 覆盖不同说法
- 有正式语气，也有口语表达
- 有直接说法，也有不直接点名但需求明显的说法

#### B. 不应触发样本
这些 prompt 应该不触发 skill。

最有价值的不是完全无关样本，而是“近邻样本”。

例如对调试 skill：

- 应触发：
  - “这个测试为什么一直挂，先帮我定位根因”
  - “先别修，帮我看下这个构建失败到底卡在哪”
- 不应触发：
  - “帮我写这个接口”
  - “给这个函数补个小改动”

### 评估重点
- 是否在近邻任务中误触发
- 是否需要更具体短语
- 是否过度依赖某个关键词

---

## 十一、建议保留的评估产物

不一定每个 skill 都要做很重的评测，但建议至少保留这些轻量产物：

### 1. 测试 prompt 列表
例如：
```json
[
  {"id": 1, "prompt": "……", "should_trigger": true},
  {"id": 2, "prompt": "……", "should_trigger": false}
]
```

### 2. 问题记录
例如一份简单 markdown：

```markdown
- Prompt 2：误触发
- Prompt 4：没有遵循“先采证”
- Prompt 5：输出缺少“下一步”
```

### 3. 版本变化摘要
每次迭代简要记一条：

- 改了什么
- 为什么改
- 想改善哪类问题

---

## 十二、推荐的轻量迭代节奏

对于 `silver_bullet` 当前阶段，不需要一开始就做复杂基准系统。  
推荐节奏：

### 第 1 轮
- 写最小可用版本
- 跑 3–5 个真实 prompt
- 修明显问题

### 第 2 轮
- 增加近邻触发样本
- 修 description
- 修边界

### 第 3 轮
- 根据真实任务再补：
  - references
  - scripts
  - examples

这样做比一上来追求“自动化完整评测平台”更实用。

---

## 十三、哪些 skill 更值得重评估

下面这些 skill 最值得认真评估：

### 1. 总控型 skill
例如：
- `silver-bullet-spec`

因为它影响的是整条复杂任务流程。

### 2. 触发边界容易模糊的 skill
例如：
- 调试类
- 分析类
- 规划类
- repo intake 类

### 3. 会引导用户进入多阶段流程的 skill
因为一旦误触发，成本很高。

---

## 十四、什么时候说明“这个 skill 已经够好了”

一个 skill 可以进入稳定可用状态，通常至少满足：

- 主要场景能触发
- 近邻场景误触发不多
- 输出结构清楚
- 没有明显关键步骤缺失
- 不需要每次都重解释它该怎么工作
- 连续两轮以上没有出现结构性问题

这不意味着它永远不改，而是说明它已经适合进入日常使用阶段。

---

## 十五、Skill 评估最小检查清单

### 目标
- [ ] 已定义 skill 想改善什么
- [ ] 已定义最小成功标准

### 测试
- [ ] 已准备 3–5 个真实质量测试 prompt
- [ ] 已准备一组触发/不触发样本
- [ ] 已至少做过一轮观察

### 分析
- [ ] 已记录触发问题
- [ ] 已记录流程问题
- [ ] 已记录内容问题

### 迭代
- [ ] 修改点与问题对应
- [ ] description 需要时已调整
- [ ] 过胖内容已拆分
- [ ] 下一轮测试样本已准备

---

## 十六、Silver Bullet 当前建议

对 `silver_bullet` 当前仓库，建议优先把评估资源放在：

1. `silver-bullet-spec`
2. `systematic-debugging-lite`

原因：

- 一个是总控 skill
- 一个是执行期专项 skill
- 它们是你当前最核心、最值得沉淀的方法资产

---

## 十七、结语

skill 评估的目的不是做复杂体系，而是避免两种常见失败：

- 只靠感觉写 skill
- 写完以后不再验证

一个好的 skill 不是一次写对的，而是在真实 prompt、真实任务和真实反馈中被不断校正出来的。

对 `silver_bullet` 来说，最重要的是建立这个迭代习惯：

- 写最小版
- 跑真实例子
- 记问题
- 改一轮
- 再跑

只要这个循环建立起来，skill 质量会比“堆很多 skill”更快提升。
