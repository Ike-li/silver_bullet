# 快速理解一个陌生代码库

写给刚入职、刚换团队、或者接手遗留系统的后端开发者。目标：在最短时间内建立全局认知，能开始干活。

---

## 为什么这很难

```
一个运行了 2 年的后端项目通常有：
  - 10 万 - 50 万行代码
  - 50 - 200 个文件
  - 10 - 30 个数据库表
  - 若干个你不熟悉的第三方库
  - 一堆没有文档的历史决策
  - 几个只有原作者知道为什么这样写的代码

从头读代码是不可能的——你会淹死在细节里。
你需要的是一套快速建立全局认知的方法。
```

---

## 第一天：建立全局地图（2-4 小时）

不要读代码。先搞清楚"这个系统是什么、做什么、怎么跑"。

### 1. 跑起来（30 分钟）

```
什么都不用理解，先让项目在本地跑起来。

步骤：
  1. 看 README → 找启动命令
  2. 看 docker-compose.yml 或 Makefile → 理解依赖什么服务
  3. 跑起来 → 打开浏览器或 Postman 访问一下
  4. 如果跑不起来 → 记下卡在哪里，问同事

跑起来比理解代码更重要。
你可以对着一个运行的系统去探索，但对着静态代码只能猜。
```

### 2. 画入口地图（30 分钟）

```
找到所有外部能访问到的入口：

  - API 路由（grep "router" 或 "@app.get" 或 "urlpatterns"）
  - 定时任务（grep "cron" 或 "celery" 或 "schedule"）
  - 消息消费者（grep "consumer" 或 "subscriber" 或 "on_message"）
  - 管理命令（grep "command" 或 "manage.py"）

快速手段：
  # FastAPI / Flask
  grep -rn "@app\.\(get\|post\|put\|delete\)" src/
  grep -rn "router\.\(get\|post\|put\|delete\)" src/

  # Django
  grep -rn "path(" */urls.py

  # 看 OpenAPI 文档（如果有）
  http://localhost:8000/docs

产出：一份 API 清单，不需要理解实现，只需要知道"有哪些入口"。
```

### 3. 看数据模型（30 分钟）

```
数据模型是理解业务最快的路径。

步骤：
  1. 找到所有 Model / 表定义
     grep -rn "class.*Model" src/
     grep -rn "CREATE TABLE" migrations/
     
  2. 画出核心实体和关系
     User → Order → OrderItem → Product
     不需要画全部，画核心的 5-10 张表就够

  3. 特别关注：
     - 哪些表数据量大（通常是业务核心表）
     - 有没有状态字段（status）→ 这些是业务状态机
     - 有没有时间字段（created_at, updated_at, deleted_at）→ 软删除？审计？

产出：一张简单的 ER 图（手画或用工具），标注核心实体和关系。
```

### 4. 看配置和依赖（30 分钟）

```
配置文件告诉你系统依赖什么：

  - pyproject.toml / requirements.txt → 依赖了什么库
  - .env.example / settings.py → 连接了什么外部服务
  - docker-compose.yml → 运行时依赖什么中间件

重点找：
  - 数据库：PostgreSQL? MySQL? MongoDB?
  - 缓存：Redis? Memcached?
  - 消息队列：Celery + RabbitMQ? Kafka? SQS?
  - 外部服务：支付? 短信? 地图? 认证?

产出：一张依赖关系图。
```

### 5. 看目录结构（15 分钟）

```
目录结构 = 代码的组织方式 = 团队的思维模型

常见的 Python 后端项目结构：

按层分（传统）：
  src/
  ├── models/       → 数据模型
  ├── services/     → 业务逻辑
  ├── api/          → 路由和控制器
  ├── schemas/      → 请求/响应格式
  └── utils/        → 工具函数

按领域分（DDD 风格）：
  src/
  ├── user/         → 用户相关的所有代码
  ├── order/        → 订单相关的所有代码
  ├── payment/      → 支付相关的所有代码
  └── shared/       → 共享代码

混合的（现实中最常见）：
  → 开始是按层分的，后来某些模块太大了拆成按领域分
  → 结果两种都有，需要你自己判断

看目录结构时问自己：
  - 新功能应该放在哪个目录下？
  - 某种类型的代码（比如数据库查询）在哪里找？
```

### 第一天结束时你应该有

```
□ 项目能在本地跑起来
□ 一份 API 入口清单
□ 一张核心数据模型图（5-10 张表）
□ 一张依赖关系图（数据库、缓存、消息队列、外部服务）
□ 对目录结构的基本认知（代码放在哪里、怎么找）
```

## 第一周：理解核心链路（每天 1-2 条）

不要试图理解所有代码。选最重要的业务链路，端到端跟一遍。

### 怎么选核心链路

```
问同事或 PM：
  "如果系统只能保证一个功能正常，保哪个？"
  
通常的答案就是核心链路：
  - 电商：下单 + 支付
  - SaaS：注册 + 核心功能使用
  - 社交：发内容 + 看内容
  - 预约系统：创建预约 + 确认预约
```

### 怎么跟一条链路

```
以"用户下单"为例：

1. 从入口开始
   找到 POST /orders 的路由定义 → 看它调了哪个函数

2. 顺着调用链往下走
   路由函数 → service 函数 → 数据库操作
   记录每一步做了什么，特别是：
   - 做了哪些校验（权限、库存、金额）
   - 操作了哪些表（orders、order_items、inventory）
   - 有没有事务（哪些操作是原子的）
   - 有没有异步操作（发消息、推通知）

3. 画出来
   POST /orders
     → check_user_auth()
     → validate_cart_items()
     → begin transaction
       → create_order()
       → create_order_items()
       → deduct_inventory()
     → commit
     → async: send_order_confirmation_email()
     → return order_id

4. 用 debugger 或日志验证
   打断点或加日志，实际走一遍这个链路，确认你的理解是对的。
```

### 用 AI 加速理解

```
把一个文件贴给 AI，问具体的问题：

✅ 好的问法：
  "这个 create_order 函数的主要流程是什么？它在哪些情况下会失败？"
  "这个中间件做了什么？什么请求会经过它？"
  "这段代码里的 retry_with_backoff 是怎么工作的？"

❌ 差的问法：
  "帮我理解这个项目"（太泛了，AI 也不知道从哪说起）
  "这个代码对吗"（没有上下文，无法判断）
```

## 第二周：理解历史和隐性知识

代码告诉你"是什么"，但不告诉你"为什么"。

### Git 历史是最好的文档

```
# 看一个文件的修改历史——理解它是怎么演变成现在这样的
git log --oneline --follow src/services/order.py

# 看某一段代码是谁在什么时候写的——找到能问的人
git blame src/services/order.py

# 看最近 3 个月改动最频繁的文件——这些是"热点"代码
git log --since="3 months ago" --name-only --format="" | sort | uniq -c | sort -rn | head -20

# 看最近的 PR 和 commit message——理解团队在做什么
git log --oneline -50
```

### 问对的人、问对的问题

```
不要问：
  "这个项目怎么回事？"（太大了，没人知道从哪开始说）
  "这段代码为什么这样写？"（可能写的人也忘了）

要问：
  "这个系统最容易出问题的地方是哪里？"
    → 直接告诉你风险区域
  "有没有哪些'不要动'的代码？"
    → 通常有历史原因，提前知道能避免踩雷
  "最近一次大的故障是什么？怎么解决的？"
    → 快速了解系统的脆弱点
  "如果重新来过，你最想改什么？"
    → 了解技术债和设计遗憾
```

### 找到那些"不在代码里的知识"

```
每个项目都有一些隐性知识，不在代码里也不在文档里：

  - "这个定时任务不能和部署同时跑，会锁表"
  - "供应商 A 的接口周末会慢 3 倍"
  - "这个表的 status=7 是历史遗留，实际已经废弃了"
  - "测试环境的数据库密码在 Confluence 的某个页面里"

这些知识通常在：
  - 老员工的脑子里
  - Slack / 飞书的历史消息里
  - 事故复盘文档里
  - 没人看的 Wiki 的某个角落

找到它们的最好方式：和团队里待得最久的人吃一顿饭。
```

## 实用技巧

### 快速定位代码的方法

```
# 知道功能但不知道代码在哪 → 从 API 入口往下找
grep -rn "POST.*order" src/

# 知道错误信息但不知道在哪抛的 → 搜错误消息
grep -rn "库存不足" src/

# 知道数据库表但不知道谁在用 → 搜表名
grep -rn "orders" src/ --include="*.py"

# 知道某个功能最近改过 → 搜 git 历史
git log --all -S "deduct_inventory" --oneline

# 想看某个函数被谁调用 → 搜函数名
grep -rn "create_order" src/ --include="*.py"
```

### 画图比记笔记有用

```
文字笔记：
  "订单服务调用库存服务扣减库存，如果失败则回滚订单，
   同时发送消息到通知队列触发邮件发送……"
  → 一段话，看完就忘

一张图：
  POST /orders
    │
    ├─ 校验 ─→ [失败] → 400
    │
    ├─ 事务 ─┬─ 创建订单
    │        ├─ 创建订单项  
    │        └─ 扣减库存 ─→ [失败] → 回滚 → 409
    │
    ├─ 异步 → 发送确认邮件
    │
    └─ 返回 order_id → 201

  → 一眼看清结构，随时能回忆起来
```

手画就行，不需要用专业工具。一张 A4 纸 + 方框 + 箭头够了。

### 建立你的"代码库导航笔记"

```
每理解一条链路就记一条，积累起来就是你的导航图：

## 核心链路
- 用户下单：POST /orders → OrderService.create() → 事务：Order + OrderItems + Inventory
- 支付回调：POST /webhooks/payment → PaymentService.handle() → 更新 Order.status
- 退款：POST /orders/{id}/refund → RefundService.create() → 异步通知支付网关

## 定时任务
- 每天 2:00：清理过期未支付订单（超过 30 分钟）
- 每小时：同步库存到搜索引擎

## 关键配置
- 支付回调超时：5 分钟内未收到回调则查询支付状态
- 库存扣减：用数据库行锁（SELECT ... FOR UPDATE）

## 踩过的坑
- order_items 表没有 order_id 的索引，查订单详情很慢（已加）
- 退款接口没有幂等校验，点两次会退两次（待修复）
```

## 时间线参考

| 时间 | 应该达到的状态 |
|:-----|:---------------|
| 第 1 天 | 项目跑起来 + 全局地图（入口、数据模型、依赖） |
| 第 1 周 | 理解 2-3 条核心链路，能做小的 bug fix |
| 第 2 周 | 理解主要模块的边界，能做小功能 |
| 第 1 个月 | 理解系统整体架构和历史决策，能独立做中等功能 |
| 第 3 个月 | 对系统有全局认知，能参与架构讨论，能指导新人 |

**不要急。** 第一个月效率低是正常的。但如果用这套方法，你会比"从头读代码"的人快 3-5 倍。

---

## 一句话总结

**不要读代码，要跟链路。** 先跑起来，再画地图，然后沿着核心业务链路从入口走到数据库。理解了 3 条核心链路，你就理解了这个系统 70% 的价值。
