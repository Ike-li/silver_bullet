# Skill 编写指南（Silver Bullet）

## 目的

本文档定义 `silver_bullet` 中 skill 的编写方法、目录结构、触发描述写法、内容组织方式，以及最小质量要求。目标不是“多写规则”，而是让每个 skill 都具备以下特征：

- 触发边界清楚
- 内容精简但可执行
- 参考资料与脚本分层清晰
- 易于维护、验证、迭代
- 不因为 skill 变多而让上下文迅速膨胀

---

## 一、什么是一个好的 skill

一个好的 skill 不是“另一段长 prompt”，而是一个可重复复用的任务包，通常包含：

- 一个明确的任务边界
- 一套稳定的处理流程
- 必要时可复用的脚本、参考文档、模板或素材
- 清晰的触发条件

它应当帮助模型在某个具体任务域里变得更稳定，而不是试图覆盖整个开发生命周期。

---

## 二、建议目录结构

推荐结构如下：

```text
skills/<skill-name>/
├── SKILL.md                 # 必需：入口文件
├── references/              # 可选：按需读取的参考文档
│   └── *.md
├── scripts/                 # 可选：可重复执行的脚本
│   ├── *.py / *.sh
│   └── requirements.txt     # 若存在脚本依赖则提供
├── examples/                # 可选：可参考的示例
│   └── *
└── assets/                  # 可选：输出需要使用但不应常驻上下文的资源
    └── *
```

### 各目录职责

#### `SKILL.md`
唯一必需文件。承担两件事：

1. 定义 skill 的元数据与触发条件
2. 给出核心流程与使用说明

#### `references/`
放详细资料、模式说明、专题指南、长文档。它们应当：

- 不常驻主上下文
- 仅在需要时被读取
- 避免和 `SKILL.md` 重复

#### `scripts/`
放确定性强、重复性高、用自然语言反复描述很浪费 token 的脚本。典型场景：

- 文件转换
- 结构化验证
- 报告生成
- 数据提取
- 固定格式修复

#### `examples/`
放“可直接参考”的样例：

- 配置样例
- 命令样例
- 输出样例
- 常见输入/输出对

#### `assets/`
放不适合加载到上下文、但会被输出流程使用的文件，例如：

- 模板
- 样张
- 图标
- 样式资源
- 基础脚手架

---

## 三、SKILL.md 的最小结构

推荐采用如下结构：

```markdown
---
name: skill-name
description: >-
  说明这个 skill 做什么，以及在什么场景下应该触发。
version: 0.1.0
---

# 标题

## 何时使用

## 不适用场景

## 核心原则

## 工作流程

## 推荐输出格式

## 附加资源
```

---

## 四、frontmatter 写法规范

### 必需字段

#### `name`
- 必须和目录名一致
- 使用小写 kebab-case
- 不要出现空格、中文、特殊符号

#### `description`
最重要的字段。它决定 skill 是否会被正确触发。

description 必须同时说明：

- 这个 skill 做什么
- 什么时候应该使用
- 哪些用户表达会触发它
- 哪些情况不该误触发

### 推荐字段

#### `version`
推荐显式写，例如：

```yaml
version: 0.1.0
```

后续 skill 演进时更容易追踪差异。

---

## 五、description 的写法要求

description 不是”简介”，而是”触发契约”。

> **关于触发倾向**：根据 Anthropic 官方观察，Claude 倾向于”不够积极地触发 skill”（undertrigger）。因此 description 应在精确和积极之间取得平衡——既要有边界，也要覆盖足够多的触发场景，包括用户没有明确提到 skill 名称但确实需要它的情况。

### 应写什么

- 具体任务类型
- 典型用户表达（多种说法，包括口语化的）
- 相邻场景的边界
- 不该触发的场景（不用于：...）
- 用户未明确说出 skill 名称但需要它的场景

### 应避免什么

- 只写抽象能力名词
- 只写”用于处理 X”
- 只写模糊词，如”复杂任务””高级工作流”
- 没有示例短语
- 触发范围过宽或过窄

### 推荐写法

```yaml
description: >-
  用于系统化定位 bug、测试失败、异常行为、构建失败与集成问题。
  当用户询问“根因是什么”“为什么会失败”“先别修先定位问题”
  或者需要复现、采证、假设验证、回归确认时触发。
  不用于普通功能开发、需求规划或单纯代码生成。
```

### 不推荐写法

```yaml
description: 提供调试帮助
```

问题：

- 太短
- 没有触发语义
- 没有边界
- 高度误触发或漏触发

---

## 六、正文写法规范

### 1. 用“客观指令”，不用“情绪化训斥”
正文应强调可执行性，不要靠口号堆压迫感。

推荐：

- 先确认复现路径，再进入修复
- 每次只验证一个假设
- 验证失败时返回上一步重新建模

不推荐：

- 绝不允许……
- 必须永远……
- 立刻停止……
- 这是失败……

### 2. 用“动词开头”的操作表达
推荐：

- 收集错误信息
- 比较成功路径与失败路径
- 运行最小验证
- 记录验证结果

不推荐：

- 你应该先……
- 你需要……
- Claude 应当……

### 3. 解释“为什么”，不要只堆规则
好的 skill 不只是告诉模型“做什么”，也会解释“为什么这样做”：

- 为什么先复现
- 为什么要单一假设
- 为什么要做回归验证
- 为什么要拆到 references 中

这样会比生硬约束更稳定。

---

## 七、渐进披露（Progressive Disclosure）

`silver_bullet` 中 skill 应遵守三层加载思路（与 Anthropic Agent Skills 规范对齐）：

### 第一层：元数据
常驻、最短、只负责触发判断。约 100 词以内。

### 第二层：`SKILL.md`
在 skill 触发时加载，负责提供核心流程、关键判断、输出格式。建议控制在 500 行以内；接近此限时应增加层级并在正文中明确指向 references。

### 第三层：资源目录
在需要时读取或执行：

- `references/`
- `scripts/`
- `examples/`
- `assets/`

### 设计原则
- `SKILL.md` 只放“必须经常读”的内容
- 详细资料移入 `references/`
- 可执行逻辑移入 `scripts/`
- 可模仿样例移入 `examples/`
- 不需要常驻上下文的文件放入 `assets/`

---

## 八、如何决定是否需要 `scripts/`

满足以下任一条件，就应优先考虑脚本而不是继续扩写 prompt：

- 同一段逻辑会反复被重写
- 结果需要确定性
- 任务本身可程序化验证
- 手工操作容易出错
- 仅靠文字说明会浪费大量 token

例子：

- 公式检查
- 数据清洗
- 格式转换
- 报告生成
- 结构校验

---

## 九、如何决定是否需要 `references/`

以下情况适合拆到 `references/`：

- 详细 API 文档
- 领域知识手册
- 复杂模式列表
- 多框架/多变体说明
- 长表格、规则集、专题说明

### 判断标准
如果某段内容不是每次触发都必读，就不要留在 `SKILL.md` 里。

---

## 十、如何决定 skill 的边界

新增 skill 之前，先问这 5 个问题：

1. 这是不是一个重复出现的任务域？
2. 它是否有稳定流程，而不是一次性 prompt？
3. 它是否和已有 skill 明显重叠？
4. 它能否独立存在，而不是依附于另一个 skill 的内部实现？
5. 它是否值得长期维护？

如果答案多数是否定的，不要急着新增 skill，先写成 prompt 或临时流程。

---

## 十一、常见反模式

### 1. 触发描述太弱
症状：

- skill 经常不触发
- 或几乎什么都能触发

### 2. `SKILL.md` 过胖
症状：

- 一份文件里塞满所有背景知识
- references 形同虚设

### 3. 资源不分层
症状：

- 样例、规则、脚本说明全写在一个文档里

### 4. 只讲理念，不讲执行
症状：

- 看起来“方法论很好”
- 但无法指导下一步动作

### 5. 和已有 skill 职责重叠
症状：

- 两个 skill 都能处理同一件事
- 总控与专项边界不清

---

## 十二、推荐长度控制

### `SKILL.md`
建议控制在“足够清楚但不臃肿”的范围内。经验上：

- 核心专项 skill：通常 100–250 行较稳
- 总控 skill：可以更长，但仍应尽量拆 references

### `references/*.md`
可以更长，但应按主题拆分，不要堆成单个超长百科文件。

---

## 十三、创建 skill 的推荐流程

1. 明确任务边界与触发场景
2. 收集 3–5 个真实或拟真的使用案例
3. 判断哪些内容应放：
   - `SKILL.md`
   - `references/`
   - `scripts/`
   - `examples/`
   - `assets/`
4. 先写一个最小可用版本
5. 设计测试 prompt
6. 用真实任务试跑
7. 根据结果调整：
   - description
   - 流程顺序
   - references 拆分
   - 是否增加脚本
8. 再进入下一轮

---

## 十四、Skill 编写最小检查清单

在提交或启用前，至少确认：

### 结构
- [ ] 目录名与 `name` 一致
- [ ] `SKILL.md` 存在
- [ ] frontmatter 合法
- [ ] `description` 具体清楚

### 内容
- [ ] 说明了何时使用
- [ ] 说明了不适用场景
- [ ] 有核心流程
- [ ] 有输出格式或交付要求
- [ ] 没有明显职责重叠

### 分层
- [ ] 详细内容已移到 `references/`
- [ ] 可程序化部分已抽到 `scripts/`
- [ ] `SKILL.md` 没有无谓膨胀

### 可维护性
- [ ] version 已设置
- [ ] 示例或参考路径可用
- [ ] 没有硬编码密钥或私密路径

## 十五、第三方 Skill 接入

从外部来源（如 `anthropics/skills`、`vendor/superpowers`、社区仓库）引入 skill 时，使用标准接入流程：

```bash
./scripts/intake-skill.sh <源skill目录> <新skill名>
```

脚本会自动：
1. 读取源 SKILL.md 的 frontmatter 和正文
2. 在 `skills/<新名>/` 下生成适配后的脚手架
3. 复制 references/scripts/examples 等资源目录
4. 生成 `INTAKE.md` 接入审计清单

### 接入后需要人工完成的工作

1. **description 适配**：补充中文触发短语和负面边界
2. **内容审计**：通读正文，确认无安全风险，裁剪不需要的部分
3. **路径修正**：所有引用路径改为适配后的路径
4. **边界审计**：确认与仓库内现有 skill 无职责重叠
5. **验证**：运行 `./scripts/validate.sh <名称>`，并用 2-3 个真实 prompt 试跑

### 接入原则

- **不要原样照搬**：第三方 skill 的 description、正文结构、资源组织可能不符合本仓库规范
- **保留来源追溯**：在 README.md 中记录原始来源、接入日期
- **只引入真正需要的**：一个 skill 解决一个问题；如果第三方 skill 太大，只提取需要的部分
- **审计优先于启用**：INTAKE.md 清单未全部勾选前，不算正式启用

---

## 十六、与 Anthropic Agent Skills 规范的对齐

本仓库的 skill 格式与 [Anthropic Agent Skills 规范](https://agentskills.io/specification) 保持兼容：

| 字段 | 官方规范 | silver_bullet |
|:-----|:---------|:-------------|
| `name` | 必需 | 必需，且必须与目录名一致 |
| `description` | 必需 | 必需，额外要求负面边界和触发短语 |
| `version` | 可选 | 推荐 |

目录结构完全兼容：`SKILL.md` + `scripts/` + `references/` + `assets/` + `examples/`。

silver_bullet 的 skill 可以直接被 Claude Code 等支持 Agent Skills 规范的工具加载。

---

## 十七、Silver Bullet 当前建议

对 `silver_bullet` 当前仓库，建议保持这类结构：

- `silver-bullet-spec`：复杂任务总控
- `systematic-debugging-lite`：执行阶段排障
- 未来新增 skill 时，优先新增“专项执行能力”，不要继续膨胀总控 skill

---

## 十八、结语

写一个 skill，不是把经验堆成一大段提示词，而是把“任务边界、触发条件、核心流程、可复用资源”组织成稳定的能力单元。

最重要的不是多，而是：

- 触发准
- 边界清
- 正文瘦
- 资源分层
- 可评估
- 可迭代
