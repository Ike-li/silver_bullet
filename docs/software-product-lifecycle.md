# 软件产品的一生——从无到有再到迭代

一个软件产品从想法到成熟，大致经历 7 个阶段。每个阶段的目标不同，团队做的事和关注点也完全不同。

## 全局视图

```
想法 → 验证 → MVP → 上线运营 → 产品市场匹配 → 规模化 → 成熟与演进
                                                          ↓
                                                    迁移 / 重写 / 下线
```

多数软件死在前三个阶段。能走到"规模化"的是少数。能优雅演进而不是被迫重写的更少。

> **贯穿案例**：以下每个阶段都用同一个产品——"一个面向中小商家的在线预约系统（BookEasy）"——来举例，展示同一个产品在不同阶段的真实决策。

---

## 阶段 0：想法与验证

**目标**：确认"这个东西值不值得做"。

### 要回答的问题

- 解决谁的什么问题？（用户是谁、痛点是什么）
- 现有方案为什么不够好？（竞品分析、差异化）
- 商业模式是什么？（怎么赚钱、或怎么度量价值）
- 技术上是否可行？（核心难点在哪、需要什么资源）

### 验证方式（从轻到重）

| 方式 | 成本 | 适用场景 |
|:-----|:-----|:---------|
| 用户访谈 | 几天 | 验证痛点是否真实存在 |
| 落地页 + 等待列表 | 1 周 | 验证有没有人关心这个问题 |
| 手动服务（Concierge MVP） | 1-2 周 | 用人工模拟软件功能，验证流程 |
| 原型 / Demo | 1-2 周 | 验证交互和核心体验 |
| 可用的 MVP | 2-6 周 | 验证用户是否真的会用 |

### 后端开发在这个阶段做什么

大多数情况下：**什么都不做**。

这个阶段的主角是产品和业务。过早投入开发是最大的浪费——做出来发现没人要，代码全扔。

但如果技术可行性是核心风险（比如"这个算法能不能跑通""这个数据量能不能实时处理"），后端需要做技术 spike（技术探针）来验证。

### 产出

- 一句话描述：给谁、解决什么问题、怎么解决
- 验证结论：做 / 不做 / 调整方向
- 如果做：初步的功能范围和优先级

### BookEasy 案例

```
想法：中小商家（美容院、健身房、诊所）还在用微信/电话接预约，经常漏单、撞单。
      做一个在线预约系统，商家发链接，客户自助约时间。

验证过程：
  1. 访谈了 12 家美容院老板 → 10 家说"有这个问题"，但只有 4 家说"愿意付费"
  2. 用石墨文档做了一个假的预约页面 + Google Form → 发给 3 家试用 1 周
  3. 结果：3 家都在用，其中 1 家主动问"正式版什么时候出，我可以付费"

验证结论：痛点真实，但付费意愿不确定（4/12）。先做 MVP 看真实付费转化。

技术 spike：无。核心功能是 CRUD + 日历，没有技术风险。
```

## 阶段 1：MVP（最小可行产品）

**目标**：用最少的代码验证核心假设。

### MVP 的核心原则

**MVP 不是"功能少的产品"，是"只包含验证假设所需最少功能的产品"。**

- 只做核心链路，砍掉所有"锦上添花"
- 只做一个用户角色的场景（不要同时做 admin + 用户 + 商家）
- 能手动的先手动（导出报表手动跑 SQL 而不是做报表系统）
- 能用第三方的先用第三方（认证用 Auth0、支付用 Stripe、邮件用 SendGrid）

### 技术决策

| 决策 | MVP 阶段的建议 |
|:-----|:---------------|
| 架构 | 单体，不拆微服务 |
| 数据库 | 一个 PostgreSQL 够用 |
| 缓存 | 不加，除非性能真的不够 |
| 消息队列 | 不加，用数据库轮询或简单的后台任务 |
| 部署 | 单机 / PaaS（Fly.io、Railway、Render），不折腾 K8s |
| 监控 | 错误追踪（Sentry）+ 基本日志，够了 |
| 测试 | 核心数据写入路径 TDD，其他手动验证 |

### 后端开发的典型工作

1. 搭建项目骨架（框架、数据库、基本中间件）
2. 设计核心数据模型（通常不超过 5-10 张表）
3. 实现核心 API（通常不超过 10-20 个端点）
4. 对接第三方服务（认证、支付、通知等）
5. 基本的错误处理和日志

### 这个阶段最大的坑

- **过度设计**：花 2 周搭"完美架构"，结果产品方向变了全推翻
- **追求代码质量**：MVP 的代码就是用来扔的。如果 MVP 验证成功，你会重写；如果失败，代码本身没有价值
- **什么都自己做**：认证、文件上传、邮件发送——这些全用第三方，把时间花在核心业务逻辑上

### 产出

- 能跑的产品（哪怕粗糙）
- 真实用户的使用数据和反馈
- 核心假设是否被验证的结论

**时长**：2-6 周。超过 6 周还没上线，大概率 scope 太大了。

### BookEasy 案例

```
MVP 范围（3 周开发）：
  ✅ 商家注册、设置可用时段
  ✅ 生成预约链接，客户打开选时间、填手机号
  ✅ 商家看到预约列表，可以确认/取消
  ✅ 客户收到短信通知

  ❌ 不做：支付（先免费用）
  ❌ 不做：多员工排班（先只支持一个人）
  ❌ 不做：微信小程序（先做 H5 网页）
  ❌ 不做：数据统计（手动导 SQL 看）

技术选型：
  - FastAPI + PostgreSQL + 部署在 Fly.io（月费 $5）
  - 短信用阿里云 SMS API（按量付费）
  - 前端用一个简单的 Vue 页面
  - 认证直接用手机号 + 验证码（不做密码体系）

数据库只有 4 张表：merchants, time_slots, bookings, sms_logs
```

## 阶段 2：上线运营与快速迭代

**目标**：产品上线后，基于用户反馈快速调整。

### 这个阶段的特点

- 需求变化极快，今天做的功能明天可能被砍
- 用户量小，技术瓶颈不是问题
- 最重要的指标是**留存率**和**用户反馈**，不是代码质量

### 后端开发的工作节奏

```
用户反馈 / 数据观察 → 定优先级 → 快速实现 → 上线 → 观察效果 → 下一轮
```

迭代周期通常 1-2 周。不需要完整的 spec-kit 流程——需求大多是"加个字段""改个逻辑""对接个新接口"。

### 需要开始建立的底线

虽然这个阶段以速度为先，但有几件事不能欠：

- **数据库 migration 版本化**：从第一天就用 migration 工具，不手动改表
- **核心数据的备份**：用户数据丢了就真丢了
- **基本的错误追踪**：Sentry 或等价物，否则你不知道线上在报什么错
- **部署自动化**：至少做到 `git push` 触发自动部署，不手动 SSH 上去部署

### 产出

- 稳定增长（或持平、或下降）的用户指标
- 明确的产品方向：哪些功能有效、哪些该砍
- 技术债务清单（知道欠了什么，但有意识地暂时不还）

### BookEasy 案例

```
上线后第一个月的真实情况：

  周 1：给之前访谈的 12 家商家发了链接，5 家注册，3 家真的在用
  周 2：美容院老板说"客户约了不来怎么办" → 加了预约前 2 小时短信提醒
  周 3：一家健身房说"我有 3 个教练，不能只支持一个人" → 记在 backlog，先不做
  周 4：1 家商家主动在朋友圈推荐了 → 新来了 8 家注册，3 家活跃

技术债务清单（有意识地欠着）：
  - [ ] 没有自动化测试（手动验证）
  - [ ] 日志是 print（看 Fly.io 控制台）
  - [ ] 没有 CI/CD（手动 fly deploy）
  - [ ] 时间冲突检测是内存锁（单机没问题，多实例会出事）
  - [ ] 短信失败没重试

最痛的 bug：两个客户约了同一个时段（并发导致），手动道歉 + 改约。
          → 在数据库加了唯一约束 (merchant_id, slot_time) 临时解决。
```

## 阶段 3：产品市场匹配（PMF）

**目标**：找到可持续增长的产品形态。

### PMF 的信号

- 用户自发推荐（NPS > 40）
- 留存率稳定（次月留存 > 30-40%）
- 明确的付费意愿或增长曲线
- 砍掉营销预算后增长不会断崖

### 技术开始需要补课

PMF 前后是技术从"先跑起来"转向"跑得稳"的拐点：

| 之前可以欠的 | 现在必须还的 |
|:-------------|:-------------|
| 没有自动化测试 | 核心链路补集成测试 |
| 没有 CI/CD | 搭建基本流水线 |
| 日志是 print | 结构化日志 + trace_id |
| 没有监控 | 基本的 metrics + 告警 |
| 单点部署 | 至少双实例 + 健康检查 |
| 手动数据库操作 | 严格走 migration |

### 后端的关键判断

这个阶段最容易犯的错是**过早优化**和**过早拆分**：

- 不要因为"以后用户多了会慢"就现在加缓存——等真的慢了再加
- 不要因为"以后会有多个服务"就现在拆微服务——等真的需要独立部署再拆
- 不要因为"以后可能换数据库"就现在抽 Repository 接口——等真的要换再抽

做最小的事，解决当下最痛的问题。

### 产出

- 明确的产品方向和商业模式
- 可以支撑下一阶段增长的技术底盘
- 团队对技术债务的清晰认知和优先级排序

### BookEasy 案例

```
第 4 个月：50 家活跃商家，12 家表示"如果有多员工排班我愿意付费"
第 6 个月：做了多员工排班 + 微信通知（替代部分短信，省钱）
第 8 个月：上了付费功能（￥99/月基础版，￥299/月专业版）
第 10 个月：30 家付费，MRR ￥6000

PMF 信号：
  ✅ 付费商家在持续增长（不靠补贴）
  ✅ 3 家商家主动发朋友圈推荐（自然增长）
  ✅ 流失率 < 10%/月
  ❌ 但还没到"爆发式增长"——增长主要靠口碑，没投广告

技术补课（这个阶段做的）：
  - 补了核心链路的集成测试（预约创建、时间冲突检测、付费扣款）
  - 搭了 GitHub Actions CI（lint + test + 自动部署到 staging）
  - 日志从 print 换成 structlog + JSON
  - 接了 Sentry 做错误追踪
  - 从单机 Fly.io 升级到双实例 + 负载均衡
```

## 阶段 4：规模化

**目标**：支撑快速增长的用户量和团队规模。

### 用户规模带来的技术挑战

| 用户量级 | 典型瓶颈 | 应对 |
|:---------|:---------|:-----|
| 1K DAU | 通常无瓶颈 | 单机够用 |
| 10K DAU | 数据库查询变慢 | 加索引、优化查询、读写分离 |
| 100K DAU | 单机扛不住 | 多实例 + 负载均衡 + 缓存层 |
| 1M DAU | 热点数据、并发冲突 | 分库分表、消息队列、异步化 |
| 10M+ DAU | 全链路瓶颈 | 微服务、多区域部署、边缘计算 |

### 团队规模带来的工程挑战

| 团队规模 | 典型问题 | 应对 |
|:---------|:---------|:-----|
| 1-3 人 | 无——口头沟通就行 | 不需要流程 |
| 5-10 人 | 代码冲突、标准不一致 | 代码规范、CI 门禁、code review |
| 10-30 人 | 模块边界不清、互相踩脚 | 服务拆分、API 契约、ownership 划分 |
| 30-100 人 | 协调成本指数增长 | 平台团队、内部框架、SLO、RFC 流程 |
| 100+ 人 | 每个决策都影响很多人 | 架构委员会、RFC 评审、渐进式迁移 |

### 这个阶段后端要做的事

1. **服务治理**：哪些服务归谁、SLO 是多少、接口契约怎么管理
2. **可观测性体系**：metrics + logs + traces 三支柱齐备
3. **容量规划**：基于增长曲线预测资源需求，提前扩容而不是等挂了再加
4. **灾备与高可用**：多可用区部署、数据库主从 + 自动切换、关键服务降级策略
5. **安全合规**：渗透测试、数据加密、审计日志、GDPR/等保合规

### 产出

- 能支撑业务增长的技术架构
- 明确的服务边界和团队 ownership
- 可量化的质量指标（SLO、error budget、P99 延迟）

### BookEasy 案例

```
第 18 个月：500 家付费商家，MRR ￥80,000，团队从 2 人扩到 8 人

用户规模挑战：
  - 周末晚高峰 QPS 从 50 涨到 500，数据库 CPU 飙到 90%
    → 加了 Redis 缓存热门商家的时段数据
    → 预约列表查询加了 (merchant_id, date) 复合索引

  - 有商家一天收到 200 个预约，列表页加载 3 秒
    → API 加了分页（之前是全量返回）

团队规模挑战：
  - 后端 3 人同时改预约模块，频繁冲突
    → 预约核心逻辑拆成独立 service 模块，定义清晰的内部 API
    → 不拆微服务（还没到那个规模），但代码层面做了模块隔离

  - 前端 2 人风格不统一
    → 引入了组件库和 design token

新增的工程实践：
  - Prometheus + Grafana 监控（QPS、延迟、错误率、DB 连接池）
  - 值班 on-call 轮转（每周一人）
  - SLO：预约接口 P99 < 500ms，可用性 99.9%
  - 部署从直接推 main 改为金丝雀发布（先给 10% 流量）
```

## 阶段 5：成熟与演进

**目标**：在保持稳定的同时持续演进。

### 这个阶段的核心矛盾

**快速迭代** vs **系统稳定**。

解决方案不是二选一，而是建立机制让两者共存：

- **Error Budget**：有预算时快速发版，预算耗尽时冻结发版优先稳定
- **Feature Flag**：新功能灰度发布，出问题秒关不需要回滚
- **渐进式迁移**：大的架构变更不做 big bang，而是新旧并行逐步切换

### 技术债务管理

到了这个阶段，技术债务是不可避免的。关键不是"消灭技术债"，而是"管理技术债"：

| 类型 | 举例 | 处理方式 |
|:-----|:-----|:---------|
| **有意识的债务** | MVP 阶段故意不做的抽象 | 排进 roadmap，分批偿还 |
| **无意识的债务** | 当时不知道更好的方案 | 发现时评估影响，决定修还是留 |
| **过时的债务** | 依赖的库/框架不再维护 | 迁移计划，渐进替换 |
| **架构级债务** | 当初的单体现在该拆了 | 最大的债，需要专项投入 |

**经验法则**：每个迭代预留 15-20% 的时间偿还技术债。不要专门拿一个月"还债"——结果通常是这个月被砍掉。

### 大规模重构与迁移

成熟系统最终会面临"要不要重写"的问题。答案几乎总是**不要重写，要渐进迁移**：

```
旧系统 → 在旧系统前加一层代理/网关
         → 新功能写在新系统
         → 旧功能逐个迁移到新系统
         → 旧系统流量归零后下线
```

这就是 Strangler Fig Pattern（绞杀者模式）。它的好处是：
- 任何时刻都可以停下来，旧系统仍在工作
- 迁移风险分散到每次小切换，而不是 big bang 上线
- 团队可以边迁移边交付新功能

### 产出

- 持续演进的系统，而不是"不敢动"的遗留系统
- 可控的技术债务水平
- 团队对系统的信心：敢改、敢发、出了问题能快速恢复

### BookEasy 案例

```
第 3 年：2000 家付费商家，MRR ￥500,000，团队 20 人

面临的矛盾：
  - 产品要快速做新功能（会员系统、营销工具、数据分析）
  - 运维要保证系统稳定（付费商家对可用性敏感——挂了就是真金白银的损失）

解决方案：
  - Error Budget 机制：99.9% SLO = 每月允许 43 分钟不可用
    有预算时快速发版（一周 3-4 次），预算耗尽时冻结发版修稳定性
  - Feature Flag：新的"会员系统"先灰度给 50 家商家，观察 2 周再全量

技术债务：
  - 最初的 4 张表现在变成了 30+ 张表，部分表有 5000 万行
  - 早期写的预约核心代码已经被改了无数次，注释和实际行为不一致
  - 处理方式：每个季度拿 20% 工时专门还债，不做"大规模重写"

架构演进（绞杀者模式）：
  - 新的"营销工具"模块作为独立服务开发（第一个微服务）
  - 通过 API 网关统一入口，外部无感知
  - 旧的预约核心不动，稳定优先
  - 新功能全部走新服务，旧功能按优先级逐步迁移
```

## 阶段 6：衰退、迁移或下线

**不是所有软件都会走到这一步，但所有软件最终都会走到这一步。**

### 触发信号

- 业务不再增长，维护成本 > 产出价值
- 技术栈过于陈旧，找不到能维护的人
- 被新产品替代（内部或外部）
- 公司战略调整，产品线收缩

### 下线不是按一个按钮

下线一个系统比上线更复杂：

1. **通知用户**：提前公告时间表（通常至少 3-6 个月）
2. **数据迁移**：用户数据迁到新系统或提供导出
3. **API 兼容**：如果有外部调用方，提供迁移指南和过渡期
4. **合规要求**：某些数据有法律保留期，不能直接删
5. **依赖解耦**：其他内部系统对它的依赖要先切走
6. **知识保留**：关键的业务逻辑和决策背景记录下来，不要随系统一起消失

## 各阶段的时间跨度参考

| 阶段 | 典型时长 | 关键指标 |
|:-----|:---------|:---------|
| 想法与验证 | 1-4 周 | 用户是否有真实痛点 |
| MVP | 2-6 周 | 能否吸引到第一批用户 |
| 上线运营 | 1-6 个月 | 留存率、用户反馈 |
| PMF | 3-18 个月 | 增长曲线、付费转化 |
| 规模化 | 1-3 年 | DAU、营收、团队规模 |
| 成熟与演进 | 3-10+ 年 | 系统稳定性、迭代速度 |
| 下线 | 3-12 个月 | 数据迁移完成度、依赖解耦 |

大多数成功的软件产品会在"成熟与演进"阶段待很久，每隔几年经历一次大的架构升级。

## 后端开发者的成长路径与各阶段的映射

| 阶段 | 后端开发者需要的能力 |
|:-----|:---------------------|
| MVP | 快速搭建、对接第三方、知道什么不做 |
| 上线运营 | 快速实现需求、基本的线上问题排查 |
| PMF | 补自动化测试、搭 CI/CD、结构化日志 |
| 规模化 | 系统设计、容量规划、服务治理、性能优化 |
| 成熟与演进 | 架构演进、渐进迁移、技术债务管理、带团队 |

每个阶段需要的能力不同。MVP 阶段的好开发是"快、能用就行"；规模化阶段的好开发是"稳、可持续、可维护"。不存在"在所有阶段都最优"的开发方式——关键是知道当前在哪个阶段，用匹配的方式做事。
