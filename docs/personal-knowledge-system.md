# 开发者的个人知识体系

写给每天都在学新东西、但总觉得"学了就忘"的技术人。

---

## 为什么你需要一个知识体系

你每天都在输入：读文档、踩坑、解 bug、review 代码、听分享、和同事讨论。但绝大多数输入在一周后消失了——不是因为你记性差，是因为没有体系承接它。

```
没有体系的状态：
  - 同样的 bug 半年后再遇到，又从头查一遍
  - "我记得之前看过一篇文章讲这个，但找不到了"
  - 面试时被问到学过的东西，但说不清楚
  - 换项目/换公司后，之前的经验好像带不走

有体系的状态：
  - 遇到类似问题，10 秒找到上次的解法和踩坑记录
  - 技术选型时，能调出之前的调研和对比结论
  - 面试或述职时，有结构化的项目复盘可以讲
  - 知识随你走，不绑定在某个公司、某个项目
```

---

## 大多数知识系统失败的原因

### 收藏癖

```
收藏了 500 篇文章，读了不到 50 篇，用到的不超过 5 篇。

收藏 ≠ 学会。收藏只是缓解了"怕错过"的焦虑，没有产生任何知识增量。
```

### 工具迷恋

```
Notion → Obsidian → Logseq → Roam Research → Heptabase → ……

每次换工具都花一周迁移数据，迁完后笔记还是一堆没人看的杂物。

工具不重要。一个 Markdown 文件夹 + 搜索，能打败 90% 精心配置的知识管理系统。
```

### 分类强迫症

```
花 2 小时给一条笔记选标签：
  这个算"数据库"还是"性能优化"？还是"Redis"？还是"缓存"？

然后这条笔记再也没被找到过。

过度分类是时间黑洞。你不是在建图书馆，你是在建工作台。
```

### 只输入不输出

```
写了 200 条笔记，从来没有一条在真实工作中被引用。

知识系统的价值不在于里面有多少内容，而在于你多频繁地使用它。
一个有 30 条高质量笔记、每周被引用 3 次的系统，
比一个有 3000 条笔记但从不打开的系统有用 100 倍。
```

---

## 核心原则：为输出而输入

知识体系的设计目标不是"存储信息"，而是"在你需要的时候，快速给你有用的东西"。

### 三个输出场景倒推

所有存入知识体系的东西，都应该为以下至少一个场景服务：

| 场景 | 你需要什么 | 举例 |
|:-----|:-----------|:-----|
| **做决策** | 之前类似问题的调研、对比、结论 | "该用 Celery 还是 RQ？我半年前调研过" |
| **解问题** | 踩过的坑、排查路径、修复方案 | "这个报错我见过，上次是连接池耗尽" |
| **教别人** | 可复用的解释、示意图、最小例子 | "新人问 Docker 网络，我有一份一页纸说明" |

如果一条笔记不服务于任何一个场景，大概率不需要记。

## 知识的四种类型

不是所有知识都值得同等对待。按"保质期"分四种：

### 1. 事实（Facts）—— 不要记

```
Python 3.12 的 match 语法是什么？Redis SET 命令的参数是什么？

这些东西 AI 和搜索引擎比你快 1000 倍。
花时间记事实是 AI 时代最大的浪费。

不记。查就行。
```

### 2. 经验（Experience）—— 必须记

```
上次 PostgreSQL 连接池耗尽的原因是什么？怎么查到的？怎么修的？

这些是你独有的、搜索引擎没有的、AI 编不出来的。
它们是你最值钱的知识资产。

必须记。而且要记过程，不只记结论。
```

### 3. 判断（Judgments）—— 值得记

```
为什么选 FastAPI 而不是 Django？在什么场景下这个判断可能是错的？

这些是你的技术决策和背后的理由。
随着经验积累，你的判断会进化——记下来可以追踪自己的成长，
也可以在类似场景下快速调用。
```

### 4. 模型（Models）—— 最值得记

```
"任何缓存问题都可以用'读路径 vs 写路径'来分析"
"分布式系统的故障排查永远从网络层开始"
"需求模糊时，先找到'谁是用户、用户的痛点是什么'，再谈方案"

这些是你从大量经验中抽象出的思维模型。
它们跨越具体技术，在任何项目、任何公司都管用。
这是高级工程师和初级工程师的真正差距。
```

### 投入分配

```
事实      ████░░░░░░░░░░░░░░░░  不记（查就行）
经验      ████████████████░░░░  重点记（独有资产）
判断      ████████████░░░░░░░░  值得记（决策参考）
模型      ████████░░░░░░░░░░░░  精心记（最高价值）
```

## 实践：一个开发者的知识体系结构

不用复杂的工具。一个 Git 仓库 + Markdown 就够了。

```

│   └── docker-dns-resolution-in-k8s.md

├── decisions/               # 判断：技术选型、架构决策
│   ├── why-fastapi-over-django.md
│   ├── when-to-split-microservices.md
│   └── celery-vs-rq-vs-dramatiq.md
```

### 每种类型的写法

#### experience（经验）

```markdown
# PostgreSQL 连接池耗尽

## 现象
2025-03-15，生产环境 API 响应时间从 50ms 飙到 10s，错误日志大量
"connection pool exhausted"。

## 排查路径
1. 看 Grafana → DB 连接数打满（100/100）
2. 查慢查询 → 没有特别慢的 SQL
3. 查应用日志 → 发现大量请求在等连接
4. 看代码 → 发现新功能有一个 API 在事务里调了外部服务（HTTP 请求）
   → 外部服务响应慢时，事务持有连接不释放 → 连接池耗尽

## 根因
事务里包了外部 HTTP 调用。外部服务变慢 → 事务变长 → 连接持有时间变长 → 连接池耗尽。

## 修复
把外部 HTTP 调用移到事务外面。先调外部服务拿到结果，再开事务写库。

## 教训
- 事务里禁止做 I/O（网络请求、文件操作）
- 连接池大小不是调大就能解决的——根因是连接持有时间
- 加了一条团队规范：code review 检查事务边界
```

关键：**记过程，不只记结论**。半年后你不记得排查路径了，但如果写下来了，5 分钟就能回忆起来。

#### decisions（判断）

```markdown
# 为什么选 FastAPI 而不是 Django

## 背景
2025-01 新项目选型，团队 3 人，做 AI 应用后端 API。

## 考虑的选项
| | FastAPI | Django | Flask |
|:--|:--------|:-------|:------|
| 异步支持 | 原生 | 3.1 开始支持但生态未完全跟上 | 需要额外配置 |
| 类型安全 | Pydantic 原生 | 需要额外引入 | 需要额外引入 |
| AI/ML 生态 | 好（异步 + 类型提示适合 LLM API 调用） | 一般 | 一般 |
| ORM | 不绑定（选了 SQLAlchemy） | 自带 Django ORM | 不绑定 |
| 学习曲线 | 低 | 中（全家桶要学的多） | 低 |

## 决策
选 FastAPI。核心理由：异步原生 + 类型安全 + 团队已有经验。

## 什么情况下这个决策是错的
- 如果需要 admin 后台 → Django 的 admin 是杀手级功能
- 如果团队不熟悉异步编程 → Django 的同步模型更安全
- 如果需要快速做完整的 Web 应用（不只是 API）→ Django 全家桶更快
```

关键：**记"什么情况下是错的"**。这比记"为什么对"更有价值——它帮你在未来场景中做更好的判断。

#### models（模型）

```markdown
# 缓存问题的读写路径分析法

## 模型
任何缓存相关的问题，都可以拆成两条路径来分析：

读路径：请求 → 缓存命中？→ 是 → 返回 / 否 → 查库 → 写缓存 → 返回
写路径：写请求 → 更新库 → 缓存怎么办？（失效 / 更新 / 不管）

## 常见问题和对应路径
| 问题 | 出在哪条路径 | 原因 |
|:-----|:-------------|:-----|
| 读到脏数据 | 写路径 | 更新库后没有失效缓存 |
| 缓存穿透 | 读路径 | 大量请求查不存在的 key，每次都打到库 |
| 缓存雪崩 | 读路径 | 大量 key 同时过期，请求涌入库 |
| 缓存击穿 | 读路径 | 热点 key 过期瞬间大量并发打到库 |

## 怎么用
遇到缓存问题时，先画出读路径和写路径，标出数据在每一步的状态。
问题一定出在某一步的状态不对——找到那一步就找到了根因。

## 来源
从 3 次缓存相关的线上故障中总结出来。
```

关键：**模型 = 你从多次经验中提炼的规律**。它的价值在于可以应用到你没见过的新场景中。

## 什么时候写、花多少时间

### 写入时机

不是每天专门留时间写笔记。而是在**这些自然节点**顺手记：

| 时机 | 记什么 | 花多长时间 |
|:-----|:-------|:-----------|
| 修完一个非平凡的 bug | experience | 10 分钟 |
| 做完一次技术选型 | decision | 15 分钟 |
| 项目上线或里程碑 | decision + experience | 30 分钟 |
| 给别人解释了一个概念 | teaches（把解释整理一下） | 10 分钟 |
| 发现自己在不同场景下用了同样的思路 | model | 20 分钟 |
| 每季度 | 更新 radar.md | 30 分钟 |

**经验法则：每周花不超过 1 小时维护知识体系。** 超过这个时间，说明你在过度记录。

### 不记的信号

- "这个搜一下就能找到" → 不记
- "这个以后可能有用" → 大概率不记（YAGNI 也适用于笔记）
- "这个很有意思" → 不记（有意思 ≠ 有用）
- "这个改变了我的理解" → 记
- "这个坑我不想再踩第二次" → 记
- "这个决策我需要向别人解释为什么" → 记

## silver_bullet 和个人知识体系的关系

你已经有了 silver_bullet，它是一个**能力资产仓库**——管理 AI skill、工作流模板、方法论文档。

它和个人知识体系的区别：

| | silver_bullet | 个人知识体系 |
|:--|:--------------|:-------------|
| 性质 | 工具和方法的集合 | 经验和判断的集合 |
| 内容 | skill、模板、通用指南 | 你的踩坑、你的决策、你的思维模型 |
| 受众 | 可以共享给团队 | 主要给自己用 |
| 生命周期 | 随工具更新 | 随你的成长积累 |

两者互补：

- silver_bullet 里的 `docs/` 是通用方法论（任何人都能用）
- 个人知识体系是你自己的实战记录（只有你经历过才有价值）
- silver_bullet 里的 `systematic-debugging-lite` 是排障方法论
- 个人知识体系里的 `experience/` 是你用这套方法论排过的真实故障

**建议**：不需要把个人知识体系放进 silver_bullet。另起一个私有仓库，它是你的私有资产。silver_bullet 是可以开源的工具箱，个人知识体系里可能有公司业务细节、团队内部的决策背景，不适合公开。

## 知识复利

知识体系的真正价值不是"查得到"，而是**复利效应**：

```
第 1 年：记了 20 条经验，做决策时偶尔翻翻
第 2 年：经验积累到 50 条，开始看到模式（"这类问题都是同一个根因"）
第 3 年：提炼出 5-10 个思维模型，遇到新问题时能快速归类
第 5 年：模型内化成直觉，看到一个设计就知道哪里会出问题

这就是"经验丰富"的真正含义——不是年限长，而是经验被结构化了。
同样干了 5 年，有体系的人积累了 5 年的复利，没体系的人重复了 5 次第 1 年。
```

---

## 一句话总结

**不要收藏，要消化。不要分类，要连接。不要追求全面，要追求在你需要时能找到。**

你的知识体系不需要漂亮、不需要完整、不需要用什么特别的工具。它需要的是：你真的在用它。
