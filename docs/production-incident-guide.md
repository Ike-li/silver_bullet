# 线上实战能力——从"能写代码"到"能扛系统"

写给能写好代码但还没经历过凌晨三点被叫醒的后端开发者。

---

## 写代码和扛系统是两种能力

```
写代码：给你一个需求，你能实现出来，测试通过，PR 合并。
扛系统：凌晨三点告警响了，你能在 15 分钟内判断问题、止血、恢复服务。

前者需要：编程能力、框架知识、测试思维。
后者需要：系统直觉、排查方法论、冷静的决策能力、对生产环境的敬畏。

大多数开发者只练了前者。但决定你薪资天花板的往往是后者。
```

---

## 故障响应的正确顺序

**止血 → 定位 → 修复 → 复盘**

不是 定位 → 修复 → 止血。

```
❌ 错误顺序（常见于经验不足的开发）：
  告警响了 → 开始看代码找根因 → 找了 30 分钟 → 发现是并发 bug
  → 写修复代码 → 测试 → 部署 → 服务恢复
  总耗时：1.5 小时，这 1.5 小时里用户一直受影响。

✅ 正确顺序：
  告警响了 → 先确认影响范围（多少用户受影响？核心链路还能用吗？）
  → 立刻止血（回滚到上个版本 / 切流量 / 开降级开关）→ 5 分钟内服务恢复
  → 然后冷静地排查根因 → 写修复 → 测试 → 部署修复版本
  → 关闭降级开关，恢复正常
  用户受影响时间：5 分钟。
```

**黄金法则：先恢复服务，再查原因。** 用户不关心你的 bug 是什么，只关心能不能用。

---

## 止血手段速查

在你开始排查之前，先评估能不能用以下手段立即止血：

| 手段 | 适用场景 | 执行时间 | 风险 |
|:-----|:---------|:---------|:-----|
| **回滚** | 最近有部署 + 问题和新版本相关 | 1-5 分钟 | 低（回到已知好的状态） |
| **重启** | 内存泄漏、进程卡死 | 1 分钟 | 中（如果根因是代码问题会复发） |
| **切流量** | 某个实例 / 可用区有问题 | 1-3 分钟 | 低 |
| **降级开关** | 非核心功能拖累核心链路 | 秒级 | 低（提前准备好的） |
| **限流** | 流量突增导致过载 | 1 分钟 | 中（会拒绝部分用户） |
| **扩容** | 正常流量增长超出容量 | 5-15 分钟 | 低 |
| **数据修复** | 脏数据导致异常 | 视情况 | 高（需要确认不引入更多问题） |

**判断逻辑**：

```
最近有部署吗？
  ├→ 有 → 回滚。80% 的故障和最近的变更有关
  │
  └→ 没有 → 资源够吗？（CPU/内存/连接池/磁盘）
              ├→ 不够 → 扩容 / 重启 / 限流
              │
              └→ 够 → 外部依赖正常吗？
                       ├→ 不正常 → 降级 / 切流量
                       │
                       └→ 正常 → 需要进入深度排查
```

## 排查方法论：从现象到根因

### 第一步：确认现象

不要听到告警就开始猜。先花 2 分钟搞清楚：

```
□ 什么指标异常？（错误率？延迟？流量？）
□ 从什么时候开始的？（精确到分钟）
□ 影响范围多大？（全部用户还是部分？哪个地区？哪个接口？）
□ 最近有什么变更？（部署、配置改动、上游变更）
```

这四个问题的答案能帮你把排查范围缩小 80%。

### 第二步：缩小范围

```
故障定位 = 不断缩小"问题可能在哪"的范围

全站挂了
  → 是所有接口还是部分接口？
    → 部分接口 → 这些接口有什么共同点？（同一个数据库？同一个下游服务？）
      → 同一个下游服务 → 那个服务怎么了？
        → 连接超时 → DNS？网络？服务本身挂了？
          → 服务本身挂了 → OOM？死锁？配置错误？
            → OOM → 找到了，某个查询返回了 100 万行数据
```

每一步都在**二分**——把"可能的原因"砍一半。

### 第三步：查证据，不查代码

```
❌ 新手的排查方式：
  "我觉得可能是这个地方有 bug" → 开始看代码 → 看了 30 分钟
  → "不对，可能是那个地方" → 又看代码 → 又看了 30 分钟

✅ 老手的排查方式：
  1. 看监控（Grafana）→ 确认什么指标、什么时间开始异常
  2. 看日志（搜 ERROR + 异常开始的时间窗口）→ 找到第一条错误日志
  3. 看链路（如果有 tracing）→ 找到出错的那个环节
  4. 到这一步才看代码 → 而且已经知道该看哪个文件的哪几行

证据链：监控 → 日志 → 链路 → 代码。不要跳步。
```

### 第四步：验证假设

```
找到"可能的原因"后，不要直接修。先验证：

假设：OOM 是因为某个查询返回了 100 万行数据
验证：
  - 看慢查询日志 → 确认有这条 SQL
  - 看该查询的执行计划 → 确认是全表扫描
  - 看该表的数据量 → 确认最近数据量暴增
  - 看是谁调了这个查询 → 确认是新上的报表功能

验证通过 → 修复（加 LIMIT / 加索引 / 加分页）
验证不通过 → 重新假设，不要在错误方向上继续
```

## 常见故障模式

后端系统 80% 的故障落在以下几类中。认识这些模式能让你排查速度翻倍。

### 1. 资源耗尽

```
表现：响应变慢 → 超时 → 报错 → 雪崩
原因：连接池满 / 内存不足 / 磁盘满 / 文件描述符耗尽 / 线程池满

排查重点：
  - 数据库连接数（SELECT count(*) FROM pg_stat_activity）
  - 内存使用趋势（是缓慢增长还是突然飙升）
  - 磁盘使用（df -h，特别注意日志目录）

典型场景：
  事务里做了外部 HTTP 调用 → 外部服务变慢 → 事务持有连接时间变长
  → 连接池耗尽 → 全站挂
```

### 2. 依赖故障

```
表现：特定功能报错，其他正常
原因：下游服务挂了 / 第三方 API 超时 / DNS 解析失败 / 证书过期

排查重点：
  - 哪个外部调用在报错（看日志或 tracing）
  - 从故障机器 curl 一下下游服务
  - 检查 DNS、证书有效期

典型场景：
  支付服务商做了一次无通知的接口升级 → 返回格式变了
  → 你的解析代码抛异常 → 所有支付失败
```

### 3. 数据问题

```
表现：业务逻辑出错，但代码没改过
原因：脏数据 / 数据倾斜 / 数据迁移遗漏 / 字符编码

排查重点：
  - 最近有没有数据迁移或批量操作
  - 出问题的记录有什么特殊的（NULL 值、特殊字符、超长内容）
  - 有没有并发写入导致的数据不一致

典型场景：
  运营在后台批量修改了 1000 个商品的价格 → 其中 3 个价格填成了 0
  → 用户下单这 3 个商品免费了 → 产生了真实的 0 元订单
```

### 4. 流量异常

```
表现：突然变慢或不可用，没有代码变更
原因：正常流量增长超出预期 / 爬虫 / 攻击 / 上游重试风暴

排查重点：
  - QPS 和正常时段对比
  - 流量来源分析（IP、UA、接口分布）
  - 上游是否在重试（一个失败触发 N 个重试 → 放大效应）

典型场景：
  某个 KOL 发了一条带链接的微博 → 10 分钟内流量涨了 50 倍
  → 数据库连接池打满 → 服务挂了
```

### 5. 配置和环境

```
表现：某个环境或某个实例有问题，其他正常
原因：配置文件错误 / 环境变量缺失 / 时区不一致 / 版本不匹配

排查重点：
  - 对比正常实例和异常实例的配置
  - 检查环境变量是否正确注入
  - 检查依赖版本是否一致

典型场景：
  新实例的时区设成了 UTC → 定时任务提前 8 小时执行
  → 凌晨 0 点跑了应该早上 8 点跑的数据清理任务 → 清掉了当天的数据
```

## 复盘：故障的价值在于不再重复

### 复盘不是追责

```
❌ "这次事故是张三写的代码有 bug，张三要反思"
   → 结果：以后大家不敢动代码，不敢上线，不敢说实话

✅ "这次事故的根因是我们的 code review 没有覆盖事务边界检查。
   Action：1) 给 review checklist 加一条；2) 加一个 lint 规则检测事务内的 I/O 调用"
   → 结果：系统变好了，所有人都受益
```

**原则：对事不对人。问"系统和流程怎么改进"，不问"谁犯了错"。**

### 复盘模板

```markdown
# 事故复盘：[简短标题]

## 概况
- 发生时间：2025-06-15 14:30 - 15:05（持续 35 分钟）
- 影响范围：全部用户，订单创建接口不可用
- 严重程度：P0

## 时间线
| 时间 | 事件 |
|:-----|:-----|
| 14:30 | 告警：订单接口 5xx 率 > 10% |
| 14:32 | 值班确认告警，开始排查 |
| 14:35 | 确认最近有部署（14:25 上线了一个新版本） |
| 14:37 | 执行回滚 |
| 14:42 | 回滚完成，5xx 率降为 0，服务恢复 |
| 14:45 | 开始排查根因 |
| 15:00 | 定位根因：新代码在事务内调用了外部地图 API |
| 15:05 | 修复代码提交，准备重新上线 |

## 根因分析
新版本在"创建订单"流程中增加了地址校验功能，
调用了外部地图 API。这个调用被放在了数据库事务内部。
地图 API 响应偶尔超过 3 秒 → 事务持有 DB 连接 3+ 秒
→ 并发请求打满连接池 → 所有数据库操作阻塞 → 订单接口全部超时。

## 为什么没有在测试/review 中发现
1. 测试环境地图 API 响应快（< 100ms），没有触发连接池瓶颈
2. Code review 没有检查事务边界（当时没有相关 checklist）
3. 没有针对"事务内 I/O"的自动化检查

## 修复措施
- [x] 立即：回滚（已完成）
- [x] 短期：把地图 API 调用移到事务外部
- [ ] 长期：review checklist 加"检查事务边界内是否有 I/O 操作"
- [ ] 长期：探索自动化 lint 规则检测事务内的网络调用

## 经验教训
- 事务内禁止做 I/O 操作（网络请求、文件读写）
- 测试环境和生产环境的外部依赖表现可能完全不同
- 回滚是最快的止血手段——前提是你的部署系统支持快速回滚
```

### 复盘的真正产出

复盘做完不是写了一份文档就结束。真正的产出是 **action items 落地**：

```
action item 的标准：
  ✅ 具体可执行："给 review checklist 加一条事务边界检查"
  ❌ 模糊无法执行："以后要更加注意代码质量"

  ✅ 有负责人和截止时间："@张三 本周五前完成"
  ❌ 无人负责："大家回去想想怎么改进"

  ✅ 可验证："添加 lint 规则，CI 能拦住事务内的 HTTP 调用"
  ❌ 不可验证："提高团队对事务管理的认识"
```

## Runbook：凌晨三点的救命稻草

Runbook 是预先写好的故障处理手册，让任何人（包括 3 AM 刚被叫醒、脑子还没转过来的你）都能按步骤操作。

### 什么时候写 Runbook

- 某类故障发生过两次以上
- 处理步骤超过 3 步
- 需要特定的命令或操作序列
- 处理者可能不是最熟悉这个系统的人（on-call 轮转）

### Runbook 模板

```markdown
# Runbook：数据库连接池耗尽

## 症状
- API 响应时间突然从 <100ms 增加到 >5s
- 错误日志出现 "connection pool exhausted" 或 "too many connections"
- Grafana 面板 "DB Connection Pool" 显示使用率 > 95%

## 影响
所有依赖该数据库的 API 不可用。

## 立即止血（5 分钟内）

### 步骤 1：确认连接池状态
psql -h <DB_HOST> -U <DB_USER> -c "SELECT count(*) FROM pg_stat_activity;"
# 如果接近 max_connections（默认 100），确认是连接池问题

### 步骤 2：找到占用连接的来源
psql -h <DB_HOST> -U <DB_USER> -c "
  SELECT client_addr, state, count(*)
  FROM pg_stat_activity
  GROUP BY client_addr, state
  ORDER BY count(*) DESC;
"
# 看哪个 IP 占用了最多连接，哪些连接是 idle in transaction

### 步骤 3：杀掉长时间空闲的事务
psql -h <DB_HOST> -U <DB_USER> -c "
  SELECT pg_terminate_backend(pid)
  FROM pg_stat_activity
  WHERE state = 'idle in transaction'
  AND state_change < now() - interval '5 minutes';
"

### 步骤 4：如果仍未恢复，重启应用
kubectl rollout restart deployment/<APP_NAME>

## 根因排查（止血后进行）
1. 查看 "idle in transaction" 连接对应的最后一条 SQL
2. 在代码中搜索该 SQL，检查事务是否正确关闭
3. 检查是否有事务内的外部调用（HTTP/RPC）
4. 检查连接池配置是否合理（pool_size, max_overflow, pool_timeout）

## 预防措施
- 设置 idle_in_transaction_session_timeout = '30s'（PG 配置）
- 应用层设置连接超时（pool_timeout = 10s）
- 监控告警：连接池使用率 > 80% 时告警
```

### Runbook 的关键原则

1. **给凌晨 3 点的人写**：不假设读者熟悉系统，每一步都要有具体命令
2. **先止血后排查**：止血步骤在前，根因排查在后，明确分开
3. **可复制粘贴**：命令直接能贴到终端跑，不需要替换变量（或明确标出需要替换的）
4. **保持更新**：系统变了 Runbook 也要更新，过时的 Runbook 比没有更危险
5. **值班时验证**：新写的 Runbook 在下次值班时演练一遍，确认能用

## On-Call 的心态

### 第一次 On-Call 前的准备清单

```
□ 能登录生产环境的跳板机 / VPN
□ 能访问监控面板（Grafana / Datadog）
□ 能访问日志系统（Kibana / Loki / CloudWatch）
□ 能执行部署和回滚（知道命令在哪）
□ 知道 Runbook 在哪
□ 知道升级路径（自己搞不定时找谁）
□ 手机能收到告警通知
□ 把以上都验证过一遍，不是"我觉得能"而是"我试过能"
```

### 处理故障时的心态

```
1. 不要慌。慌了判断力下降，反而更慢。
   深呼吸，按流程来。

2. 不要一个人扛。
   如果 10 分钟内定位不了，立刻拉人。
   "我搞不定"不丢人，让用户多挂 30 分钟才丢人。

3. 记录你做了什么。
   开一个文档或者在群里实时记录。
   复盘时最有价值的材料就是时间线。

4. 止血优先于完美修复。
   先让服务恢复，哪怕方法不优雅。
   "回滚"不是失败，是最成熟的止血手段。

5. 故障后不要带着情绪。
   每次故障都是系统变好的机会，不是谁的过错。
```

## 建立你的线上能力

线上能力不是靠经历事故练出来的（虽然事故确实是最好的老师），而是可以主动建设的：

| 行动 | 什么时候做 | 收获 |
|:-----|:-----------|:-----|
| **读别人的复盘报告** | 每周花 30 分钟 | 见识不同类型的故障模式 |
| **做故障演练** | 每季度 | 在可控环境中练习排查和止血 |
| **写 Runbook** | 做完一个功能时顺手写 | 积累操作手册 |
| **值班时主动排查** | on-call 期间 | 真实环境练手 |
| **看监控面板** | 每天花 5 分钟 | 建立"正常状态"的基线感觉 |

最后一条特别重要：**你必须知道"正常是什么样"，才能发现"异常"。** 每天看一眼 Grafana，看多了你会对那些曲线形成直觉——不用看数字就知道今天是不是正常。

---

## 一句话总结

**线上能力 = 止血速度 × 排查方法论 × 心理素质。** 三者都可以在没有事故的时候提前练。等到凌晨三点再学，就晚了。
