# 技术写作——让别人愿意读、读得懂、读完能做决策

写给觉得"写文档浪费时间"的后端开发者。技术写作不是文学创作，是工程活动的一部分。

---

## 为什么技术人需要会写

```
你写的代码会被其他开发读。
你做的技术决策需要被团队理解。
你遇到的故障需要被复盘。
你推动的方案需要被领导批准。

这些全都需要写作——不是"写得好"，是"写得清楚"。

一份好的技术文档能：
  - 让一个异步讨论代替一个 1 小时会议
  - 让新人自助理解系统，而不是每次都找你问
  - 让三个月后的你自己记住"当时为什么这样做"
  - 让领导在 5 分钟内做出决策，而不是听你讲 30 分钟

一个写不清楚的技术方案 = 一个需要反复开会澄清的技术方案。
写作不是浪费时间，是节省所有人的时间。
```

## 技术写作的核心原则

### 1. 读者优先，不是作者优先

```
❌ 作者视角（按你想的顺序写）：
  "我先调研了 A，然后看了 B，发现 C 有个问题，
   所以考虑了 D 和 E，最后决定用 F……"
  → 读者要跟着你的思考过程走一遍，很累

✅ 读者视角（按读者需要的顺序写）：
  "方案：用 F。原因：满足 X 需求且成本最低。
   对比了 D 和 E，D 不满足 Y，E 成本高 3 倍。详见下文。"
  → 读者 10 秒就知道结论，有兴趣再看细节
```

**规律：结论先行，细节在后。** 大多数读者只需要结论和理由，不需要你的完整思考过程。

### 2. 一件事说一次

```
❌ 同一个信息在文档的三个地方出现（概述里说一次、详情里说一次、总结里说一次）
  → 读者不知道以哪个为准，改的时候容易漏

✅ 每个信息只在最合适的地方出现一次，其他地方引用
  → 改一处就够了
```

### 3. 可扫读

```
技术文档不是小说，没有人从头到尾逐字读。

要做到：
  - 标题层级清晰，看目录就知道结构
  - 关键结论加粗
  - 用列表而不是长段落
  - 用表格做对比而不是用文字描述
  - 代码块有语言标记和必要注释

读者应该能在 30 秒内通过标题和加粗文字得到核心信息。
有兴趣再深入看细节。
```

### 4. 精确

```
❌ 模糊："系统性能会变好"
✅ 精确："P99 延迟从 800ms 降到 200ms"

❌ 模糊："大概要几天"
✅ 精确："预计 3 个工作日，其中 1 天写代码、1 天写测试、1 天联调"

❌ 模糊："可能会有并发问题"
✅ 精确："当两个请求同时更新同一行时，后提交的会覆盖先提交的（丢失更新）"
```

## 四种核心文档及其写法

### 一、RFC / 技术方案（决策文档）

**目的**：让团队对一个技术决策达成共识。

**关键**：读完后，每个人都能回答"做什么、为什么、怎么做、有什么风险"。

```markdown
# RFC: [标题——一句话说清楚要做什么]

## 状态
Draft / In Review / Accepted / Rejected / Superseded by RFC-XXX

## 背景
[为什么要做这个？现在有什么问题？]
用 2-3 句话描述现状和痛点。用数据支撑，不用感受。

## 目标
[做了之后达到什么效果？]
- 目标 1（可量化）
- 目标 2（可量化）

**非目标**（明确不解决的问题）：
- 非目标 1
- 非目标 2

## 方案

### 推荐方案：[方案名]
[用 3-5 句话描述方案的核心思路]

**数据模型变更**：
[如果有表结构变化，列出来]

**API 变更**：
[如果有接口变化，列出来]

**架构图**：
[如果有，画一个简单的方框图]

### 备选方案：[方案名]
[简要描述]
**不选的原因**：[1-2 句话]

### 备选方案：[方案名]
[简要描述]
**不选的原因**：[1-2 句话]

## 风险和缓解
| 风险 | 可能性 | 影响 | 缓解措施 |
|:-----|:-------|:-----|:---------|
| ... | 高/中/低 | 高/中/低 | ... |

## 工作量估算
| 任务 | 预估 | 负责人 |
|:-----|:-----|:-------|
| ... | X 天 | @谁 |

## 开放问题
[还没想清楚的、需要讨论的问题]
1. ...
2. ...
```

**写 RFC 的常见错误**：

```
1. 没有"非目标"
   → 范围无限膨胀，review 时大家往里面塞需求
   → 写清楚"不做什么"和"做什么"一样重要

2. 只写推荐方案，不写备选
   → 别人不知道你考虑过什么、排除了什么
   → 至少列 1-2 个备选 + 不选的原因

3. 风险部分是空的或敷衍的
   → "没有风险"= 你没想过
   → 每个方案至少有 1 个风险。写出来，加缓解措施

4. 工作量估算缺失
   → 决策者不知道投入产出比，没法批
```

### 二、设计文档（实现细节）

**目的**：给执行者（包括未来的你）一份可参考的技术蓝图。

**与 RFC 的区别**：RFC 回答"做不做、选哪个方案"；设计文档回答"具体怎么做"。

```markdown
# 设计文档：[功能名]

## 概述
[一段话描述要实现什么]

## 数据模型
[表结构、字段说明、索引策略]

## 接口设计
[API 路径、请求/响应格式、错误码]

## 核心流程
[用流程图或伪代码描述主要逻辑]

## 边界情况
[列出需要特殊处理的场景]
- 并发下单同一商品怎么办？
- 用户余额不足但已经扣了库存怎么办？
- 外部支付回调延迟或丢失怎么办？

## 测试策略
[哪些需要单元测试、哪些需要集成测试]

## 监控和告警
[上线后需要关注什么指标、什么情况触发告警]

## 部署计划
[是否需要数据库迁移、是否需要灰度、回滚方案]
```

### 三、事故报告（复盘文档）

**目的**：记录发生了什么、为什么、怎么防止再发生。

见 `production-incident-guide.md` 中的完整复盘模板。这里只强调写作要点：

```
1. 时间线必须精确到分钟
   不是"下午出了个问题"，是"14:32 收到告警，14:35 确认影响范围"

2. 根因要写到"为什么没有被提前发现"
   不只是"代码有 bug"，而是"code review 没有检查事务边界"

3. action items 必须可执行、有 owner、有截止日期
   不是"加强代码质量意识"，是"@张三 本周五前给 review checklist 加一条"

4. 语气对事不对人
   不是"张三的代码导致了故障"，是"某段代码的事务边界设计不合理"
```

### 四、ADR（架构决策记录）

**目的**：记录"为什么这样做"，让后来的人不会推翻一个有充分理由的决策。

```markdown
# ADR-001: 使用 PostgreSQL 而不是 MySQL

## 状态
Accepted（2025-03-15）

## 背景
新项目需要选择关系型数据库。

## 决策
使用 PostgreSQL。

## 理由
1. 需要 JSONB 字段存储灵活属性（MySQL 的 JSON 性能差一个量级）
2. 团队有 PostgreSQL 运维经验
3. 需要的扩展功能（全文搜索、地理位置）PostgreSQL 原生支持

## 后果
- 正面：JSONB 查询性能好，减少了 join
- 负面：团队中 2 人需要从 MySQL 切换到 PostgreSQL（预计 1 周适应）
- 风险：运维工具链需要确认是否完全支持（已确认）

## 被否决的方案
- MySQL：JSONB 性能不足
- MongoDB：不需要完全放弃关系模型，且团队无经验
```

**ADR 的关键**：短。一页以内。不是论文，是决策记录。

## 写作效率技巧

### 先写结构，再填内容

```
不要打开文档就开始写。先花 5 分钟列提纲：

1. 这篇文档的读者是谁？
2. 读完后他们应该能做什么？
3. 他们最需要知道的 3 件事是什么？

然后按重要程度排列，写出标题结构。
标题结构确定后，每个标题下的内容自然就知道写什么了。
```

### 写完后删掉 30%

```
初稿写完后，重读一遍，问自己：
  - 这段话删掉后，文档的核心信息会丢失吗？
  - 这个段落是在帮助读者理解，还是在展示我知道很多？
  - 这句话能不能用一半的字说清楚？

通常初稿有 30% 是可以删掉的。删掉后文档反而更好读。
```

### 让别人读一遍

```
写完后找一个不了解背景的人读一遍，问他：
  - "你能用一句话概括这篇文档在说什么吗？"
  - "有没有哪里看不懂？"
  - "读完后你知道下一步该做什么吗？"

如果他概括不出来，说明你的结构有问题。
如果他看不懂某一段，说明那段写得太隐晦。
如果他不知道下一步该做什么，说明你的 action 不够明确。
```

### 用 AI 辅助写作

```
AI 适合做的：
  - 帮你整理思路："我想写一个关于 X 的 RFC，要点是 1/2/3，帮我列个提纲"
  - 帮你精简文字："帮我把这段话缩短到一半，保留核心信息"
  - 帮你检查遗漏："这个技术方案有什么我可能遗漏的风险？"
  - 帮你统一格式："把这些要点整理成表格"

AI 不适合做的：
  - 替你做决策（AI 不知道你的业务上下文）
  - 替你写"为什么"（原因在你脑子里，不在 AI 的训练数据里）
  - 替你理解读者（你知道领导关心什么，AI 不知道）
```

## 什么时候不需要写文档

不是所有东西都需要写文档。过度文档和没有文档一样有害。

```
不需要文档的：
  - 代码本身就能说清楚的（好的命名 > 注释 > 文档）
  - 一次性的小改动（commit message 就够了）
  - 标准的 CRUD 接口（API 自描述 + OpenAPI 就够了）
  - 团队内部心照不宣的约定（如果所有人都知道，不需要写下来）

需要文档的：
  - 有多个方案需要选择的（RFC）
  - 设计决策的理由不在代码中的（ADR）
  - 新人需要了解的系统概览（设计文档）
  - 出了故障需要复盘的（事故报告）
  - 操作步骤需要精确执行的（Runbook）

判断标准：如果三个月后有人（可能是你自己）会问"当时为什么这样做"，
就值得写下来。如果不会，代码和 commit message 就够了。
```

## 一句话总结

**好的技术文档不是写得多，是每一份都在正确的时间帮正确的人做出正确的决策。** 结论先行、精确、可扫读——做到这三点就超过了 90% 的技术文档。
