# 系统设计思维——面对真实需求时怎么想

不是面试八股文，是你拿到一个需求后脑子里应该跑的思考过程。

---

## 系统设计不是画架构图

```
大多数人理解的系统设计：
  → 画一个方框图，左边客户端，中间负载均衡，右边数据库
  → 中间加几个微服务方框，用箭头连起来
  → 标注"用 Redis 缓存""用 Kafka 解耦""用 K8s 部署"
  → 觉得自己设计完了

真正的系统设计：
  → 理解需求的边界（什么必须做、什么可以不做）
  → 估算规模（数据量、流量、并发——决定了方案的量级）
  → 做取舍（一致性 vs 可用性、性能 vs 成本、简单 vs 灵活）
  → 从最简单的方案开始，识别瓶颈，逐步演进

架构图是结果，不是过程。过程是一连串的判断和取舍。
```

---

## 第一步：问对问题

拿到需求后不要直接想方案。先问清楚边界。

### 功能边界

```
需求："做一个短链接服务"

要问的：
  - 谁用？内部系统还是对外公开？
  - 需要自定义短码吗？还是只要随机生成？
  - 短链接有过期时间吗？
  - 需要统计点击量吗？统计到什么粒度？（总量 / 按天 / 按地区）
  - 需要防滥用吗？（有人用你的服务生成钓鱼链接怎么办）

每一个问题的答案都会改变技术方案。
"自定义短码"意味着需要唯一性校验；
"统计按地区"意味着需要解析 IP；
"防滥用"意味着需要 URL 安全扫描。

在搞清楚这些之前画的任何架构图都是猜的。
```

### 规模边界

```
同样的需求，规模不同方案完全不同：

每天 1000 个短链接 → 单机 + SQLite 就够了
每天 100 万个短链接 → 需要考虑 ID 生成策略、数据库分片
每天 1 亿个短链接 → 需要考虑分布式 ID、多区域部署、缓存层

估算方法：
  日活用户 × 每用户操作次数 = 日请求量
  日请求量 ÷ 86400 ≈ 平均 QPS
  平均 QPS × 峰值倍数（通常 3-10 倍）= 峰值 QPS

  日请求量 × 每条数据大小 = 日数据增量
  日数据增量 × 365 × 保留年数 = 总存储量
```

### 非功能边界

```
□ 延迟要求：用户能接受多慢？（短链接跳转 < 100ms？报表查询 < 3s？）
□ 可用性要求：能接受多少停机时间？（99.9% = 每月 43 分钟）
□ 一致性要求：能接受读到旧数据吗？（缓存可以有几秒延迟？）
□ 成本约束：预算是多少？（决定了你能用什么服务）
□ 安全要求：数据敏感吗？需要加密吗？合规要求？
```

## 第二步：从最简单的方案开始

**不要一上来就画微服务架构。** 先想最简单的方案能不能满足需求。

### 实践：短链接服务

```
最简单的方案（单机、单表）：

  客户端 → FastAPI → PostgreSQL
  
  表设计：
  CREATE TABLE short_urls (
    id SERIAL PRIMARY KEY,
    short_code VARCHAR(8) UNIQUE NOT NULL,
    original_url TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT NOW(),
    click_count INTEGER DEFAULT 0
  );
  
  短码生成：base62(自增 ID)
  跳转：SELECT original_url WHERE short_code = ?
  
  够用吗？日均 1000 个短链接 + 每天 10 万次跳转 → 单机绰绰有余。
```

然后问自己：**这个方案的瓶颈在哪？什么时候会不够用？**

```
瓶颈分析：

1. 跳转查询是读密集的 → 如果 QPS 到 10000，数据库扛不住
   解法：加 Redis 缓存（短码 → 原始 URL），缓存命中率会极高

2. 自增 ID 暴露了信息（能猜到你总共生成了多少） 
   解法：用随机短码代替 base62(自增 ID)

3. 单点故障 → 一台机器挂了服务就没了
   解法：两个实例 + 负载均衡

4. 统计 click_count 在高并发下有热点更新问题
   解法：异步计数（写入 Redis 原子 INCR，定期刷到数据库）

演进后的方案：
  客户端 → 负载均衡 → FastAPI (×2) → Redis (缓存 + 计数) → PostgreSQL
```

### 演进思维

```
不要一步到位。按这个节奏：

V1：最简单的方案，满足当前需求
  → 上线，观察真实数据

V2：解决 V1 中已经出现的瓶颈
  → 上线，观察

V3：解决 V2 中已经出现的新瓶颈
  → 上线，观察

每一步都是因为有真实的问题才演进，不是因为"将来可能有问题"。

"将来可能有问题"的东西有无数个。
只有"现在已经有问题"的东西值得你花时间。
```

## 第三步：识别核心取舍

每个设计决策都是取舍。好的系统设计不是没有缺点，是缺点在你能接受的地方。

### CAP 不是背的，是用的

```
CAP 定理说：一致性、可用性、分区容忍性不能同时满足。

在实际工作中，分区（网络故障）是无法避免的，
所以真正的选择是：网络出问题时，你选一致性还是可用性？

选一致性（CP）：
  银行转账 → 宁可拒绝服务，也不能出现"钱扣了但没到账"
  库存扣减 → 宁可下单失败，也不能超卖

选可用性（AP）：
  社交媒体的点赞数 → 少一个赞不会死人，但页面加载不出来会流失用户
  商品浏览记录 → 少记一条没关系，但服务挂了影响推荐

大多数系统不是纯 CP 或纯 AP：
  同一个电商系统中：
  - 支付链路选 CP（宁可慢，不能错）
  - 商品详情页选 AP（宁可展示缓存的旧价格，不能打不开）
  - 用户头像选 AP（旧头像显示几秒没关系）
```

### 常见取舍清单

| 取舍 | 左边 | 右边 | 怎么选 |
|:-----|:-----|:-----|:-------|
| 一致性 vs 性能 | 强一致（每次读都是最新的） | 最终一致（读到的可能是旧的） | 涉及钱和权限选左，其他选右 |
| 简单 vs 灵活 | 硬编码业务规则 | 可配置的规则引擎 | 规则 < 5 条选左，> 20 条且频繁变选右 |
| 实时 vs 批量 | 每条数据实时处理 | 攒一批统一处理 | 用户在等结果选左，内部报表选右 |
| 自建 vs 买 | 自己实现 | 用第三方服务 | 非核心竞争力的选右 |
| 成本 vs 性能 | 省钱（小实例、少冗余） | 花钱买性能和可靠性 | 取决于停机一小时的损失和你的预算 |

## 第四步：数据模型先行

**数据模型决定了系统 80% 的上限。** API 可以改、代码可以重构、架构可以演进，但数据模型改起来代价最大。

### 设计数据模型的思考过程

```
需求：电商的订单系统

1. 识别核心实体
   → 用户(User)、商品(Product)、订单(Order)、订单项(OrderItem)

2. 确定关系
   → User 1:N Order（一个用户多个订单）
   → Order 1:N OrderItem（一个订单多个商品）
   → Product 1:N OrderItem（一个商品出现在多个订单中）

3. 确定每个实体的关键属性
   → Order：状态(status)、总金额(total_amount)、创建时间、支付时间
   → OrderItem：数量(quantity)、单价(unit_price)、小计(subtotal)

4. 问自己这些问题：
   - 哪些字段需要做查询条件？→ 需要索引
   - 哪些数据会频繁更新？→ 要考虑锁和并发
   - 哪些数据需要历史记录？→ 要不要做快照

5. 关键决策：
   Q：OrderItem 里要不要冗余商品名称和价格？
   A：要。因为商品可能改价或下架，但订单记录应该反映下单时的价格。
      这是"快照"而不是"冗余"——它们的语义不同。

   Q：订单金额是实时计算还是存储的？
   A：存储。下单时算好存进去。原因：优惠规则可能变，但已下单的金额不应该变。
```

### 数据模型的常见陷阱

```
1. 用外键关联代替数据快照
   ❌ OrderItem 只存 product_id → 商品改价后，历史订单的金额对不上
   ✅ OrderItem 存 product_id + unit_price 快照

2. 用枚举代替状态机
   ❌ status VARCHAR → 谁都能改成任意值
   ✅ 明确定义合法的状态转换，代码里校验

3. 软删除到处用
   ❌ 所有表都加 is_deleted 字段 → 所有查询都要加 WHERE NOT is_deleted
   ✅ 只在需要恢复或审计的表用软删除，其他直接删

4. 过度范式化
   ❌ 用户地址拆成 province、city、district 三张表
   ✅ 除非你需要按省/市维度做聚合查询，否则一个 JSON 字段就够

5. 没有考虑时间维度
   ❌ 会员等级只存当前值 → 查不到"这个用户上个月是什么等级"
   ✅ 如果业务需要历史记录，设计时就加 effective_from / effective_to
```

## 第五步：识别风险点

设计完方案后，主动找"这个方案哪里会出事"。

### 风险清单模板

```
对每个方案问自己：

□ 单点故障：哪个组件挂了整个系统就挂了？
□ 数据丢失：断电、崩溃时数据会不会丢？
□ 热点问题：有没有某个数据/接口被不成比例地高频访问？
□ 数据倾斜：会不会所有数据集中在一个分片/分区？
□ 级联故障：一个服务挂了会不会连锁导致其他服务也挂？
□ 容量天花板：当前方案最多支撑多大的量？超过了怎么办？
□ 运维复杂度：这个方案需要多少运维投入？团队有能力维护吗？
```

### 实践：秒杀系统的风险识别

```
需求：商品秒杀，1000 件商品，预计 10 万人抢

风险分析：
  1. 热点问题：所有请求打到同一行数据（库存字段）→ 行锁排队 → 数据库垮
     应对：库存预加载到 Redis，用 DECR 原子扣减，数据库只做最终确认

  2. 级联故障：秒杀流量打垮整个系统，影响正常购物
     应对：秒杀独立部署，和主站隔离

  3. 超卖：并发扣减导致库存变负数
     应对：Redis DECR 后检查是否 < 0，是则 INCR 回去 + 返回失败

  4. 前端重复提交：用户狂点按钮
     应对：前端按钮点一次后禁用 + 后端幂等（用 user_id + activity_id 去重）

  5. 机器人/脚本抢购：不公平
     应对：接口限流（同一 IP 1 秒最多 1 次）+ 验证码
```

## 设计模式速查：常见场景的方案

不是让你背方案，是给你一个起点。具体细节要根据你的规模和约束调整。

### 读多写少（商品详情、用户资料）

```
方案：应用 → 缓存 → 数据库
缓存策略：Cache-Aside（先查缓存，miss 了查库并回填缓存）
失效策略：写入时失效缓存 + TTL 兜底
注意：缓存穿透（查不存在的 key）→ 缓存空值或布隆过滤器
```

### 写多读少（日志、埋点、IoT 数据）

```
方案：应用 → 消息队列 → 消费者 → 数据库/数据仓库
为什么不直写库：写太频繁，数据库扛不住
消息队列作用：削峰填谷，异步批量写入
注意：消息可能重复消费 → 消费者要幂等
```

### 读写都多（社交 feed、实时聊天）

```
方案取决于数据模型：
  推模式（写扩散）：发消息时写入每个接收者的收件箱 → 读快写慢
  拉模式（读扩散）：读的时候从发送者那里拉 → 写快读慢
  推拉结合：活跃用户用推，不活跃用户用拉
注意：没有完美方案，根据用户行为分布选
```

### 全局唯一 ID 生成

```
方案（从简单到复杂）：
  1. 数据库自增 → 最简单，单点瓶颈
  2. UUID → 无协调，但无序且占空间
  3. Snowflake → 有序、分布式，但依赖时钟
  4. 号段模式 → 批量从数据库取一段 ID，本地分配
选哪个：QPS < 1000 用 1，需要有序用 3 或 4，不需要有序用 2
```

### 分布式锁

```
场景：多个实例需要互斥访问同一资源
方案：
  1. 数据库行锁 → 最简单，性能不高
  2. Redis SETNX + 过期时间 → 主流方案，注意续期问题
  3. Redlock → Redis 多节点分布式锁，争议大但常用
  4. ZooKeeper / etcd → 最可靠，运维成本高
选哪个：大多数场景 Redis SETNX 够用。金融级用 etcd。
```

## 一句话总结

**系统设计不是选技术栈，是做取舍。** 好的设计来自于对需求的深刻理解、对规模的准确估算、以及"先简单后演进"的纪律。
