# 基于后端接口文档的测试执行指南

写给大模型或测试执行 Agent：当你拿到一份后端接口文档时，按照本文档的方法开始测试。

目标不是简单地“把接口都请求一遍”，而是从接口契约、业务链路、异常路径、权限、状态、幂等、并发和数据一致性等角度，系统性地发现接口可能失败的地方。

---

## 输入材料

执行测试前，至少需要以下材料：

- 后端接口文档
- 测试环境地址
- 认证方式说明
- 可用测试账号及其角色
- 测试数据库或测试数据准备方式
- 业务流程说明或核心用户路径
- 约定的测试工具，例如 pytest、Postman、Apifox、curl、requests、locust

如果材料不完整，不要直接脑补。先输出“待确认问题清单”，把缺失信息列出来。

---

## 总体执行流程

拿到接口文档后，按以下顺序执行：

1. 审查接口文档完整性
2. 按业务链路理解接口
3. 识别高风险接口
4. 设计接口测试矩阵
5. 先做接口契约测试
6. 再做功能和异常路径测试
7. 补充权限、状态、幂等、并发和数据一致性测试
8. 对核心链路做 E2E 或冒烟测试
9. 输出测试报告和风险清单

不要一开始就写自动化测试代码。先把测试范围、测试重点和缺失信息梳理清楚。

---

## 第一步：审查接口文档完整性

逐个接口检查文档是否包含以下信息：

| 检查项 | 说明 |
|:--|:--|
| 接口名称 | 这个接口解决什么业务问题 |
| URL | 完整路径，例如 `/api/v1/orders/{id}` |
| Method | GET、POST、PUT、PATCH、DELETE |
| 认证方式 | 是否需要 token、cookie、签名、API key |
| 权限角色 | 哪些角色可以访问 |
| Header | 必传 header、可选 header、Content-Type |
| Path 参数 | 参数名、类型、是否必填、示例 |
| Query 参数 | 参数名、类型、是否必填、默认值、枚举值 |
| Body 参数 | 字段结构、类型、必填、长度限制、格式限制 |
| 成功响应 | 状态码、响应字段、字段类型、示例 |
| 失败响应 | 参数错误、未登录、无权限、资源不存在、业务冲突等 |
| 错误格式 | 错误码、错误消息、错误字段是否统一 |
| 业务规则 | 状态流转、金额计算、库存扣减、唯一性约束等 |
| 幂等性 | 重复请求是否允许，是否需要 Idempotency-Key |
| 分页规则 | page/page_size 或 cursor，默认值和上限 |
| 排序和过滤 | 支持哪些字段，非法字段怎么处理 |
| 限流规则 | 高频接口是否有限流 |
| 外部依赖 | 是否依赖支付、短信、邮件、第三方服务 |

如果某个字段没有写清楚，记录为待确认问题。

示例输出：

```markdown
## 文档待确认问题

- `POST /api/v1/orders` 没有说明库存不足时返回 400、409 还是 422。
- `GET /api/v1/orders` 没有说明 `page_size` 最大值。
- `POST /api/v1/payments` 没有说明是否支持幂等请求。
- 错误响应示例不统一，有的使用 `error`，有的使用 `message`。
```

---

## 第二步：按业务链路理解接口

不要只孤立地看接口。先把接口放进用户真实操作流程里。

例如订单系统：

```text
注册/登录
  -> 查询商品
  -> 创建订单
  -> 支付订单
  -> 查询订单状态
  -> 取消订单/申请退款
```

对每条链路识别：

- 入口接口是什么
- 哪些接口会修改数据
- 哪些接口依赖前置状态
- 哪些接口会影响金额、库存、权限、账户余额
- 哪些接口失败后必须保证数据一致
- 哪些接口是上线后必须冒烟验证的核心路径

输出示例：

```markdown
## 核心业务链路

### 下单链路

1. `POST /api/v1/login`
2. `GET /api/v1/products/{id}`
3. `POST /api/v1/orders`
4. `POST /api/v1/payments`
5. `GET /api/v1/orders/{id}`

关键风险：

- 创建订单和扣库存必须一致。
- 支付接口必须幂等。
- 支付失败时订单不能变成已支付。
- 用户不能访问其他用户的订单。
```

---

## 第三步：识别高风险接口

优先测试高风险接口。以下接口需要重点覆盖：

| 接口类型 | 风险 |
|:--|:--|
| 登录、注册、鉴权 | 账号安全、越权、暴力破解 |
| 创建订单、支付、退款 | 金额错误、重复扣款、状态不一致 |
| 库存扣减 | 超卖、并发竞争 |
| 用户资料、地址、隐私数据 | 数据泄露、越权访问 |
| 管理后台接口 | 权限提升、误删数据 |
| 文件上传 | 文件类型绕过、大小限制、安全风险 |
| 搜索、筛选、排序 | SQL 注入、性能问题 |
| Webhook、回调接口 | 签名校验、重复通知、伪造请求 |
| 批量操作接口 | 部分成功、事务一致性、权限混用 |

低风险接口可以轻测，高风险接口必须深测。

---

## 第四步：设计接口测试矩阵

为每个接口建立测试矩阵，明确要测哪些维度。

| 接口 | 契约 | 正常流程 | 参数校验 | 权限 | 状态流转 | 幂等 | 并发 | 数据一致性 | 外部依赖 |
|:--|:--|:--|:--|:--|:--|:--|:--|:--|:--|
| 登录 | 是 | 是 | 是 | 不适用 | 不适用 | 可选 | 可选 | 可选 | 否 |
| 创建订单 | 是 | 是 | 是 | 是 | 是 | 是 | 是 | 是 | 是 |
| 支付订单 | 是 | 是 | 是 | 是 | 是 | 是 | 是 | 是 | 是 |
| 查询订单 | 是 | 是 | 是 | 是 | 可选 | 不适用 | 不适用 | 可选 | 否 |
| 取消订单 | 是 | 是 | 是 | 是 | 是 | 是 | 是 | 是 | 可选 |

执行原则：

- 每个接口至少要有契约测试。
- 写接口必须测权限和数据一致性。
- 涉及金额、库存、账户余额的接口必须测并发和幂等。
- 涉及状态机的接口必须测合法转换和非法转换。
- 依赖第三方服务的接口必须测超时、失败和异常返回。

---

## 第五步：契约测试

契约测试用于验证接口是否符合文档约定。

每个接口至少检查：

- 正确请求返回正确状态码
- 响应 JSON 结构正确
- 字段类型正确
- 必填字段存在
- 可空字段行为明确
- 时间格式正确
- 金额精度正确
- 枚举值在允许范围内
- 错误响应格式统一
- 不返回敏感字段
- Content-Type 正确

示例：

```python
def test_get_order_contract(client, auth_header):
    resp = client.get("/api/v1/orders/1", headers=auth_header)

    assert resp.status_code == 200
    assert resp.headers["content-type"].startswith("application/json")

    data = resp.json()
    assert isinstance(data["id"], int)
    assert isinstance(data["status"], str)
    assert isinstance(data["created_at"], str)
    assert "password_hash" not in data
    assert "internal_flags" not in data
    assert "traceback" not in data
```

错误响应也必须做契约测试：

```python
def test_error_response_contract(client, auth_header):
    resp = client.get("/api/v1/orders/999999999", headers=auth_header)

    assert resp.status_code == 404

    data = resp.json()
    assert "error" in data
    assert "message" in data
    assert "traceback" not in data
```

## 第六步：功能和异常路径测试

对每个接口按以下分类设计测试用例。

### 正常场景

- 必填参数都合法
- 可选参数不传时使用默认值
- 可选参数传入合法值
- 枚举字段传入每个合法枚举值
- 分页、排序、过滤正常工作

### 参数异常

- 缺少必填字段
- 字段类型错误
- 字段为空字符串
- 字段为 null
- 字段超长
- 数值为 0
- 数值为负数
- 数值超过最大值
- 枚举值非法
- 日期格式非法
- JSON 格式错误
- Content-Type 错误

### 数据边界

- 最小合法值
- 最大合法值
- 刚好小于最小值
- 刚好大于最大值
- 空列表
- 超大列表
- 重复元素
- 特殊字符
- Unicode 字符
- SQL 注入字符串
- XSS 字符串

示例参数集：

```text
""
" "
null
"a" * 10000
"'; DROP TABLE users;--"
"<script>alert(1)</script>"
"用户😀"
"\n\t"
```

---

## 第七步：权限测试

权限测试必须覆盖以下场景：

- 未登录访问需要登录的接口
- token 为空
- token 无效
- token 过期
- 普通用户访问管理员接口
- A 用户访问 B 用户资源
- 被禁用用户访问接口
- 删除后的用户继续访问接口
- 角色权限变更后旧 token 是否仍然有效

示例：

```python
def test_user_cannot_access_other_users_order(client, user_a_header, user_b):
    order = create_order(user_id=user_b.id)

    resp = client.get(f"/api/v1/orders/{order.id}", headers=user_a_header)

    assert resp.status_code in (403, 404)
    assert "traceback" not in resp.text
```

注意：如果产品要求隐藏资源存在性，可以返回 404；如果要求明确权限不足，可以返回 403。以接口文档和安全策略为准。

---

## 第八步：状态流转测试

如果接口操作的是有生命周期的业务对象，必须先画出状态机。

订单示例：

```text
pending -> paid -> shipped -> delivered
pending -> cancelled
paid -> refunded
shipped -> returned
```

需要测试：

- 每条合法状态转换
- 每条非法状态转换
- 重复执行同一个转换
- 失败后状态是否保持不变
- 状态变更后相关数据是否同步更新

非法转换示例：

```text
cancelled -> paid
delivered -> cancelled
pending -> shipped
refunded -> shipped
```

---

## 第九步：幂等测试

以下接口必须考虑幂等性：

- 创建订单
- 支付
- 退款
- 取消订单
- Webhook 回调
- 批量操作
- 任何可能被客户端重试的写接口

需要测试：

- 同一个请求发送两次
- 同一个 Idempotency-Key 发送两次
- 同一个业务单号发送两次
- 第一次成功，第二次重试
- 第一次超时，客户端重试
- 并发发送两个相同请求

断言重点：

- 不重复扣款
- 不重复创建资源
- 不重复发券
- 不重复发消息
- 返回结果一致或明确说明已处理

示例：

```python
def test_payment_is_idempotent(client, auth_header):
    order = create_order(amount=100)
    key = "test-payment-key-001"

    resp1 = client.post(
        "/api/v1/payments",
        json={"order_id": order.id},
        headers={**auth_header, "Idempotency-Key": key},
    )
    resp2 = client.post(
        "/api/v1/payments",
        json={"order_id": order.id},
        headers={**auth_header, "Idempotency-Key": key},
    )

    assert resp1.status_code == 200
    assert resp2.status_code == 200
    assert resp1.json()["transaction_id"] == resp2.json()["transaction_id"]
    assert count_payments(order.id) == 1
```

## 第十步：并发和数据一致性测试

以下场景必须测并发：

- 抢库存
- 支付
- 退款
- 账户余额变更
- 优惠券领取
- 唯一资源创建
- 批量审批

断言重点：

- 库存不能为负
- 余额不能被重复扣减
- 唯一资源不能重复创建
- 成功数量符合预期
- 失败请求有明确错误
- 数据库最终状态正确

示例：

```python
def test_concurrent_stock_deduction():
    product = create_product(stock=1)

    results = run_concurrent(
        lambda: buy_product(product.id, user_id=1),
        lambda: buy_product(product.id, user_id=2),
    )

    success_count = sum(1 for r in results if r.success)
    assert success_count == 1
    assert get_stock(product.id) == 0
```

## 第十一步：外部依赖故障测试

如果接口依赖外部系统，必须模拟故障。

常见故障：

- 第三方超时
- 第三方返回 500
- 第三方返回 200 但 body 非法
- 第三方返回业务失败
- 第三方重复回调
- 第三方签名错误
- 网络中断

断言重点：

- 系统不能挂死
- 不应该返回未处理的 500
- 错误响应应清晰
- 本地业务状态不能错误推进
- 后续重试机制符合预期

---

## 第十二步：E2E 和冒烟测试

E2E 不追求覆盖所有分支，只覆盖核心链路。

每个系统选择 3 到 5 条最关键链路：

- 用户注册到首次登录
- 下单到支付成功
- 支付失败后重新支付
- 订单取消或退款
- 管理员审核或配置生效

冒烟测试用于上线后快速确认：

- 服务可访问
- 登录可用
- 核心查询可用
- 核心写入可用
- 核心异步任务或回调可用

## 执行注意事项

- 不要只测 happy path。
- 不要把所有接口都当成同等优先级。
- 不要在文档缺失时自行决定业务规则。
- 不要只验证状态码，还要验证数据库最终状态。
- 不要只验证接口单点，还要验证核心业务链路。
- 不要用真实第三方服务做常规自动化测试，应该使用 mock、stub 或测试沙箱。
- 不要让测试之间依赖执行顺序。
- 不要留下 flaky test。

最终判断标准：不是“接口都请求成功了”，而是“核心业务链路已验证，关键失败路径已覆盖，已知风险都有明确测试或待确认问题”。
