# Systematic Debugging Lite（系统化调试轻量版）

这是一个**独立调试 skill**，目标是帮助你在修复前先查清根因，避免“猜测式修复”和补丁叠补丁。

## 何时使用

- “这个 bug 的根因到底是什么？”
- “测试为什么挂了？”
- “这次改动后为什么行为异常？”
- “构建失败 / 集成失败是在哪一层断的？”
- “先别修，先定位根因。”

## 不适用场景

- 普通功能开发与需求实现
- 常规规划、任务拆解、进度管理
- 仅需代码生成、并无调试问题
- 根因已明确且只差执行一个极小修复动作

## 核心原则

1. **先根因，后修复**：未完成根因调查，不给修复结论。
2. **先证据，后结论**：结论必须能被日志、堆栈、复现、对比结果支持。
3. **一次只验证一个假设**：避免多变量同时修改导致不可归因。
4. **连续失败要升级**：连续多次修复失败后，回到架构/方案层讨论，不继续盲目堆 patch。

## 轻量调试流程（5 阶段）

### Phase 1：问题确认与复现

**目标**：确认“是什么问题”，并拿到可重复触发路径。  
**动作**：

1. 逐行阅读错误信息、堆栈、日志与报错上下文。
2. 明确复现步骤、输入、环境、频率（是否稳定复现）。
3. 若无法复现，先补采证据，不直接给修复建议。

**产出**：
- 问题描述（现象 + 触发条件）
- 可复现步骤（或明确“暂不可稳定复现”与缺失证据）

### Phase 2：证据采集与边界定位

**目标**：确定问题发生在系统的哪一层/哪一段边界。  
**动作**：

1. 检查近期改动（代码、配置、依赖、环境差异）。
2. 多组件链路按边界采证：输入是什么、输出是什么、在哪一步开始异常。
3. 收集“工作路径 vs 失败路径”差异，不先下结论。

**产出**：
- 已知证据清单
- 可疑边界与排除项

### Phase 3：假设与最小验证

**目标**：用最小实验验证单一根因假设。  
**动作**：

1. 写清楚当前假设：`我认为 X 是根因，因为 Y 证据`。
2. 设计最小验证动作（最小改动、最小输入、最短反馈）。
3. 仅验证一个变量；若失败，废弃该假设并重建，不叠加改动。

**产出**：
- 当前假设
- 验证动作
- 验证结果（支持 / 反驳 / 不确定）

### Phase 4：修复与回归验证

**目标**：只修根因，并确认修复有效且未引入回归。  
**动作**：

1. 优先准备最小复现用例或失败测试（脚本/测试皆可）。
2. 执行单点修复，不顺手做无关重构。
3. 运行对应验证：原问题不再出现 + 相关测试/构建通过。

**产出**：
- 修复说明（修了什么、为何有效）
- 回归验证结果

### Phase 5：连续失败后的升级处理

**触发信号**：同一问题修复后立即引入新回归；或同一根因假设反复被推翻却找不到替代假设；或已尝试多轮修复但症状未收敛。
**动作**：

1. 暂停继续打补丁。
2. 评估是否为架构耦合、边界设计、方案前提错误。
3. 回流到上层流程（如 `silver-bullet-spec` 的分析/计划）讨论方案调整。

**产出**：
- 升级结论（为何需要架构/计划层处理）
- 建议的下一步（重拆任务、改边界、调整方案）

## 明确禁止

- 上来就拍脑袋改代码
- 一次修改多个变量并一起验证
- 没有复现就宣布定位完成
- 没有验证就宣布修复完成
- 连续失败还继续堆 patch

## 推荐输出格式（每轮调试）

请尽量结构化输出：

1. **现象**：发生了什么、影响范围  
2. **已知证据**：日志/堆栈/对比/复现信息  
3. **当前假设**：本轮只保留一个主要假设  
4. **验证动作**：本轮要执行的最小验证  
5. **验证结果**：支持/反驳/不确定 + 关键输出  
6. **下一步**：继续验证、实施修复或升级讨论

## 与 `silver-bullet-spec` 的协作

- `silver-bullet-spec` 负责复杂任务总控（分析、计划、进度、归档）。
- `systematic-debugging-lite` 负责执行阶段问题排查（定位、证据、假设验证、修复前门禁）。
- 当执行阶段出现 bug/test failure/unexpected behavior 时，先用本 skill；若确认是架构级问题，再回流 `silver-bullet-spec` 做方案层调整。
