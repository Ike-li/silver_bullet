# S.U.P.E.R 架构哲学

> 像搭乐高一样写代码：每块只做一件事、接口标准、流向清晰、随处可跑、随时可换。

本文定义在 **silver-bullet-spec** 工作流中指导分析、规划与实施的架构原则。执行任务者应内化这些原则。

---

## S — 单一职责（Single Purpose）

源自 Unix 哲学。

- 每个模块、文件、函数只解决一个问题
- 倾向拆分；组合产生能力
- 一个 skill 一件事，一个工人一件事，一个脚本一件事

**试金石：** 若无法用一句话说清模块职责，就需要拆分。

**反例：** 一个脚本既拉数据、又算指标、又画图、又发通知。

**正例：**
```
fetch_data.py  -> 只拉数据，输出 JSON
compute.py     -> 只计算，读 JSON 写 JSON
render.py      -> 只渲染，读 JSON 出 HTML
notify.py      -> 只通知，读 JSON 调 webhook
```

---

## U — 单向流动（Unidirectional Flow）

源自整洁架构。

- 数据单向流动：输入 → 处理 → 输出
- 依赖始终向内：外层依赖内层，内层不感知外层
- 禁止反向依赖与循环调用

**分层示意：**
```
+-------------------------------+
|  基础设施（API、DB、UI）       |  <- 最外层，可随意替换
+-------------------------------+
|  适配器（转换、格式化）        |
+-------------------------------+
|  核心业务（纯逻辑）            |  <- 最内层，零外部依赖
+-------------------------------+
```

**试金石：** 核心逻辑能否在零外部服务的情况下跑单元测试？若不能，依赖方向有问题。

---

## P — 端口优于实现（Ports over Implementation）

源自六边形架构。

- 先定义接口契约（数据结构、JSON Schema），再写实现
- 用中间格式（JSON、标准数据结构）隔离上游与下游
- 替换数据源、渲染层、通知渠道时，核心逻辑零改动

**实践：**
1. 每个模块的输入输出必须是可序列化的数据结构
2. 模块边界通过 JSON 或标准数据结构通信；进程内类型对象可以，跨模块接口必须可序列化
3. 定义显式 schema，而非「读代码猜格式」

---

## E — 与环境无关（Environment-Agnostic）

源自 12-Factor。

- 配置经环境变量或配置文件注入，禁止硬编码
- 依赖在清单文件中显式声明（requirements.txt / package.json 等），不隐式依赖全局包
- 进程无状态；持久化交给外部存储
- 日志走标准输出，不写死路径文件
- 同一套代码可在本机、Cloudflare Workers、VPS、Docker 等环境运行

**配置优先级（高 → 低）：**
```
环境变量 > .env 文件 > config.json > 代码内默认值
```

**自检：**
- API 密钥、Webhook 是否都从环境变量读取？
- 依赖是否都在依赖清单中声明？
- 是否避免硬编码路径假设？
- 换一台机器零改代码能否运行？

---

## R — 可替换部件（Replaceable Parts）

S + U + P + E 的自然结果与终极目标。

- 任一层可替换而不牵连其他层
- **替换成本**是架构质量的核心指标
- 若替换一个组件导致无关模块连锁修改，则架构已坏

**替换矩阵：**
| 替换对象 | 影响范围 | 正确做法 |
|:---------|:---------|:---------|
| 数据源 API | 仅适配层 | 新抓取器，输出相同 JSON |
| 前端渲染 | 仅渲染层 | 读相同 JSON，换实现 |
| 通知渠道 | 仅通知层 | 换 webhook 适配器 |
| 部署平台 | 仅部署配置 | 改 wrangler.toml 或 Dockerfile 等 |
| 编程语言 | 仅实现层 | JSON 契约不变，语言可重写 |

---

## 快速检查卡

```
+------------------------------------------+
|         S.U.P.E.R 快速检查                |
|                                          |
|  S  该模块是否只做一件事？                |
|  U  数据流是否单向？                     |
|  P  输入输出是否有 schema/契约？        |
|  E  换环境能否跑？                       |
|  R  替换时是否不会波及无关模块？         |
|                                          |
|  全是 -> 架构健康                       |
|  1～2 否 -> 需要重构                    |
|  3+ 否 -> 技术债告警                    |
+------------------------------------------+
```
